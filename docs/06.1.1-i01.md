# I01 — Proporção de Servidores por Regime de Trabalho

## 1. Finalidade do indicador

O I01 calcula a **distribuição percentual dos servidores com Planos de Trabalho ativos no período entre os regimes de trabalho cadastrados no PETRVS** — tipicamente Presencial, Híbrido e Teletrabalho.

A pergunta central que o I01 responde é: **"Como a força de trabalho do ICMBio está distribuída entre os regimes de atuação no período analisado?"**

O indicador tem duas variantes de resultado complementares:

- **Variante geral (institucional):** uma linha por regime, com o total e o percentual de servidores da instituição em cada modalidade. Responde à pergunta no nível do ICMBio como um todo.
- **Variante por unidade:** uma linha por combinação regime × unidade, com o percentual de servidores em cada regime calculado dentro da própria unidade. Responde à pergunta no nível de cada área.

Diferente dos demais indicadores deste manual, o I01 não mede desempenho — mede a condição estrutural em que o trabalho é realizado. Seu valor principal é fornecer contexto para a interpretação dos Eixos 2, 3 e 4: uma unidade com 80% dos servidores em teletrabalho integral opera com dinâmica diferente de uma unidade 80% presencial, e essa diferença é relevante ao comparar taxas de cumprimento, distribuição de carga ou notas de avaliação.

> **Pré-requisito:** antes de executar, confirme os nomes dos regimes cadastrados no seu banco com as consultas de mapeamento do [Eixo 1](06.1-eixo1.md). Os nomes exatos da coluna `tipos_modalidades.nome` variam entre instâncias do PETRVS.

## 2. Consultas completas

### Variante 1 — Proporção geral (visão institucional)

```sql
with parametros as (
    select
        CAST('2025-01-01' AS DATE) as data_inicio,
        CAST('2025-12-31' AS DATE) as data_fim,
        0                          as incluir_excluidos
),
servidores_no_periodo as (
    select distinct
        pt.usuario_id,
        coalesce(tm.nome, 'N.I.') as modalidade
    from planos_trabalhos pt
    left join tipos_modalidades tm
        on tm.id = pt.tipo_modalidade_id
    cross join parametros p
    where CAST(pt.data_inicio AS DATE) <= p.data_fim
      and CAST(pt.data_fim AS DATE) >= p.data_inicio
      and (p.incluir_excluidos = 1 or pt.deleted_at is null)
      and pt.usuario_id is not null
),
contagem_por_regime as (
    select
        modalidade,
        count(distinct usuario_id) as total_servidores
    from servidores_no_periodo
    group by modalidade
)
select
    modalidade,
    total_servidores,
    round(
        total_servidores * 100.0
            / nullif(sum(total_servidores) over (), 0),
        2
    ) as proporcao_perc
from contagem_por_regime
order by total_servidores desc;
```

### Variante 2 — Proporção por regime dentro de cada unidade

```sql
with parametros as (
    select
        CAST('2025-01-01' AS DATE) as data_inicio,
        CAST('2025-12-31' AS DATE) as data_fim,
        0                          as incluir_excluidos
),
servidores_por_unidade as (
    select distinct
        coalesce(un.sigla, 'N.I.') as unidade_sigla,
        coalesce(un.nome,  'N.I.') as unidade_nome,
        pt.usuario_id,
        coalesce(tm.nome,  'N.I.') as modalidade
    from planos_trabalhos pt
    left join tipos_modalidades tm
        on tm.id = pt.tipo_modalidade_id
    left join unidades un
        on un.id = pt.unidade_id
    cross join parametros p
    where CAST(pt.data_inicio AS DATE) <= p.data_fim
      and CAST(pt.data_fim AS DATE) >= p.data_inicio
      and (p.incluir_excluidos = 1 or pt.deleted_at is null)
      and pt.usuario_id is not null
),
contagem as (
    select
        unidade_sigla,
        min(unidade_nome)           as unidade_nome,
        modalidade,
        count(distinct usuario_id)  as total_servidores
    from servidores_por_unidade
    group by unidade_sigla, modalidade
)
select
    unidade_sigla,
    unidade_nome,
    modalidade,
    total_servidores,
    round(
        total_servidores * 100.0
            / nullif(sum(total_servidores) over (partition by unidade_sigla), 0),
        2
    ) as proporcao_na_unidade_perc
from contagem
order by unidade_sigla, total_servidores desc;
```

## 3. Passos da consulta

A Variante 1 e a Variante 2 compartilham a mesma arquitetura de três CTEs. A explicação abaixo descreve os blocos da Variante 1 com as diferenças da Variante 2 assinaladas em cada passo.

### Passo 1 — Bloco `parametros`: painel de controle

```sql
parametros as (
    select
        CAST('2025-01-01' AS DATE) as data_inicio,
        CAST('2025-12-31' AS DATE) as data_fim,
        0                          as incluir_excluidos
)
```

Idêntico ao padrão dos demais indicadores. O período de análise filtra planos de trabalho cuja vigência se sobrepõe ao intervalo definido — um plano que começou em setembro/2024 e termina em março/2025 seria incluído numa análise de 2025.

Para comparar a distribuição de regimes entre dois períodos (ex: primeiro e segundo semestre do ano), execute a consulta duas vezes com intervalos distintos.

---

### Passo 2 — Bloco `servidores_no_periodo`: pares únicos servidor × regime

Este é o bloco mais importante do I01. Ele identifica, para cada servidor com plano ativo no período, em qual regime ele estava trabalhando.

```sql
servidores_no_periodo as (
    select distinct
        pt.usuario_id,
        coalesce(tm.nome, 'N.I.') as modalidade
    from planos_trabalhos pt
    left join tipos_modalidades tm
        on tm.id = pt.tipo_modalidade_id
    cross join parametros p
    where CAST(pt.data_inicio AS DATE) <= p.data_fim
      and CAST(pt.data_fim AS DATE) >= p.data_inicio
      and (p.incluir_excluidos = 1 or pt.deleted_at is null)
      and pt.usuario_id is not null
)
```

**A estrutura dos joins:**

- `from planos_trabalhos pt`: ponto de partida — o plano de trabalho de cada servidor. Cada linha contém `usuario_id` (o servidor), `tipo_modalidade_id` (o regime) e as datas de vigência.
- `left join tipos_modalidades tm on tm.id = pt.tipo_modalidade_id`: busca o nome textual do regime. O `left join` garante que planos sem regime cadastrado (`tipo_modalidade_id` nulo) não sejam perdidos — eles aparecem com modalidade `'N.I.'` graças ao `coalesce(tm.nome, 'N.I.')`.
- `cross join parametros p`: inclui os parâmetros de configuração para que os filtros de data e exclusão sejam aplicados.

**O filtro de período:**

```sql
where CAST(pt.data_inicio AS DATE) <= p.data_fim
  and CAST(pt.data_fim AS DATE) >= p.data_inicio
```

Esse par de condições seleciona todos os planos cuja vigência tem qualquer sobreposição com o período de análise — não apenas planos que começaram e terminaram dentro do intervalo. Um plano de outubro/2024 a junho/2025 seria incluído numa análise do ano de 2025, pois parte dele cobre o período.

**O `select distinct`:** é a proteção central deste bloco. Um servidor pode ter mais de um plano de trabalho ativo no período (ex: um plano encerrou em março e outro começou em abril). Sem o `distinct`, o servidor apareceria duas vezes. Com o `distinct` no par `(usuario_id, modalidade)`, cada combinação servidor-regime aparece **uma única vez**, independentemente de quantos planos sustentam essa combinação.

**Comportamento de dupla contagem intencional:** se um servidor mudou de regime durante o período — por exemplo, estava em Híbrido no primeiro semestre e migrou para Teletrabalho no segundo —, ele terá dois planos com regimes diferentes, e o `select distinct` produzirá **duas linhas**: uma para Híbrido e uma para Teletrabalho. Esse comportamento é deliberado e correto: o servidor efetivamente atuou nos dois regimes no período. A seção iv detalha como lidar com isso na interpretação.

**Na Variante 2**, este bloco adiciona duas colunas — `unidade_sigla` e `unidade_nome` — e o join com `unidades`. A lógica de filtragem e o `select distinct` são idênticos.

---

### Passo 3 — Bloco `contagem_por_regime`: agregando servidores por regime

```sql
contagem_por_regime as (
    select
        modalidade,
        count(distinct usuario_id) as total_servidores
    from servidores_no_periodo
    group by modalidade
)
```

Agrupa os pares do passo anterior por regime e conta quantos servidores distintos estão em cada modalidade.

O `count(distinct usuario_id)` é uma camada adicional de proteção: mesmo que o bloco anterior já tenha aplicado `select distinct`, o `distinct` no `count` garante que servidores que aparecem em múltiplas linhas por terem mudado de regime dentro do mesmo regime (situação rara mas possível por inconsistência de cadastro) não sejam contados mais de uma vez por modalidade.

**Na Variante 2**, o `group by` é `(unidade_sigla, modalidade)` — produzindo uma linha por combinação unidade × regime. O `min(unidade_nome)` retorna o nome da unidade para exibição, pois `unidade_nome` não faz parte do `GROUP BY` mas é funcionalmente dependente de `unidade_sigla` (uma unidade tem sempre o mesmo nome).

---

### Passo 4 — SELECT final: calculando a proporção com window function

#### Variante 1 — proporção sobre o total institucional

```sql
select
    modalidade,
    total_servidores,
    round(
        total_servidores * 100.0
            / nullif(sum(total_servidores) over (), 0),
        2
    ) as proporcao_perc
from contagem_por_regime
order by total_servidores desc;
```

A coluna central é `proporcao_perc`. Ela divide o total de servidores de cada regime pelo total geral de servidores da instituição no período.

**A window function `sum(total_servidores) over ()`:** este é o recurso que torna a operação possível sem uma subconsulta. O `over ()` sem cláusula `PARTITION BY` significa "calcule a soma sobre todas as linhas do resultado". Cada linha recebe o mesmo denominador — o total geral — enquanto mantém seu próprio numerador. Isso permite calcular o percentual de cada regime em relação ao total em uma única passagem sobre os dados.

**Analogia:** é como se você tivesse uma planilha com o total de servidores por regime e quisesse adicionar uma coluna de percentual. Em vez de calcular o total em uma célula separada e referenciar nela, a window function "enxerga" todas as linhas ao mesmo tempo e usa a soma delas como denominador automaticamente.

**O `nullif(..., 0)`:** proteção contra divisão por zero — se por algum motivo não houver servidores no resultado, retorna `NULL` em vez de erro.

#### Variante 2 — proporção dentro de cada unidade

```sql
round(
    total_servidores * 100.0
        / nullif(sum(total_servidores) over (partition by unidade_sigla), 0),
    2
) as proporcao_na_unidade_perc
```

A diferença em relação à Variante 1 está no `PARTITION BY unidade_sigla`. Enquanto na Variante 1 o denominador é o total de servidores de toda a instituição, aqui o denominador é **o total de servidores da própria unidade**. O `PARTITION BY` recria o cálculo separadamente para cada unidade — cada servidor é comparado apenas com os colegas da sua unidade, não com toda a instituição.

**Exemplo do efeito do `PARTITION BY`:** se a CGOV tem 20 servidores (15 em teletrabalho, 5 em híbrido) e a AUDIT tem 10 servidores (8 presencial, 2 híbrido):

| Unidade | Regime | Total | Sem `PARTITION BY` | Com `PARTITION BY` |
| --- | --- | --- | --- | --- |
| CGOV | Teletrabalho | 15 | 15/30 = 50% | 15/20 = 75% |
| CGOV | Híbrido | 5 | 5/30 = 17% | 5/20 = 25% |
| AUDIT | Presencial | 8 | 8/30 = 27% | 8/10 = 80% |
| AUDIT | Híbrido | 2 | 2/30 = 7% | 2/10 = 20% |

Sem `PARTITION BY`, a proporção é sobre os 30 servidores totais — útil para a Variante 1. Com `PARTITION BY`, a proporção é sobre o tamanho da própria unidade — útil para a Variante 2, onde o objetivo é entender o perfil de cada unidade independentemente do seu tamanho.

## 4. Como interpretar o resultado

### Estrutura do resultado

**Variante 1 — Proporção geral:**

| Coluna | O que informa |
| --- | --- |
| `modalidade` | Nome do regime conforme `tipos_modalidades.nome` |
| `total_servidores` | Servidores distintos com plano ativo no período neste regime |
| `proporcao_perc` | Percentual do total institucional de servidores com planos ativos |

**Variante 2 — Proporção por unidade:**

| Coluna | O que informa |
| --- | --- |
| `unidade_sigla` / `unidade_nome` | Identidade da unidade |
| `modalidade` | Regime de trabalho |
| `total_servidores` | Servidores distintos da unidade neste regime |
| `proporcao_na_unidade_perc` | Percentual dentro da própria unidade |

**Como interpretar os perfis esperados para cada unidade (baseado na natureza da atividade):**

**CGOV (Coord. de Governança):** unidade de planejamento e regulação em Brasília — espera-se predominância de teletrabalho e/ou híbrido. Ao cruzar com o I02 (execução), avaliar se a proporção de teletrabalho está correlacionada com a taxa de cumprimento de entregas.

**AUDIT (Auditoria Interna):** unidade de controle interno — pode apresentar maior proporção presencial por acesso a documentos físicos e sistemas restritos. Ao cruzar com o I05 (carga), verificar se o regime influencia a distribuição de entregas por servidor.

**COGEO (Coord. de Geoinformação):** unidade técnica — perfil típico de híbrido ou teletrabalho para analistas, mas com possível componente presencial para atividades de campo e laboratório.

**Unidades de campo (UC — Parques Nacionais, Reservas, Florestas Nacionais):** regime presencial dominante reflete a natureza da atividade fim — gestão de território, fiscalização, manejo. Comparar essas unidades com unidades de governança usando os mesmos thresholds de I02 requer ajuste pelo contexto de regime.

### A dupla contagem intencional — entendendo o fenômeno

Quando um servidor muda de regime durante o período analisado, ele aparece em duas linhas do resultado — uma para cada regime em que teve plano ativo. Isso faz com que a **soma das proporções possa ultrapassar 100%** na Variante 1.

Exemplo: se 20 servidores mudaram de Híbrido para Teletrabalho em julho de 2025, e o período de análise é o ano inteiro:
- Eles aparecem tanto na linha "Híbrido" quanto na linha "Teletrabalho";
- O total de linhas do resultado será maior que o número real de servidores distintos.

Esse comportamento é **metodologicamente correto para análise de regime vigente no período**. Para uma visão alternativa que elimine a dupla contagem — mostrando apenas o **regime mais recente de cada servidor** —, adicione o filtro de snapshot antes do bloco `contagem_por_regime`:

```sql
-- Snippet para obter apenas o regime mais recente por servidor
-- Inserir entre 'servidores_no_periodo' e o SELECT final
regime_atual as (
    select
        usuario_id,
        modalidade,
        row_number() over (
            partition by usuario_id
            order by data_inicio desc
        ) as rn
    from (
        select distinct
            pt.usuario_id,
            coalesce(tm.nome, 'N.I.') as modalidade,
            pt.data_inicio
        from planos_trabalhos pt
        left join tipos_modalidades tm
            on tm.id = pt.tipo_modalidade_id
        cross join parametros p
        where CAST(pt.data_inicio AS DATE) <= p.data_fim
          and CAST(pt.data_fim AS DATE) >= p.data_inicio
          and (p.incluir_excluidos = 1 or pt.deleted_at is null)
          and pt.usuario_id is not null
    ) base
)
-- Substituir 'servidores_no_periodo' por: select usuario_id, modalidade from regime_atual where rn = 1
```

A versão com `ROW_NUMBER()` produz proporções que somam exatamente 100%, porque cada servidor aparece em apenas um regime.

### O que significa a categoria `N.I.`

A categoria `'N.I.'` (Não Informado) agrupa todos os planos de trabalho onde `tipo_modalidade_id` está nulo — ou seja, o regime não foi cadastrado. Algumas interpretações possíveis:

- **Planos antigos** criados antes de o campo de modalidade ser obrigatório no PETRVS
- **Planos migrados** de versões anteriores do sistema, onde o campo não existia
- **Erro de preenchimento** — o servidor ou gestor não informou o regime ao cadastrar o plano

Se `N.I.` representar mais de 10% do total, o resultado do I01 deve ser apresentado com a ressalva de que uma parcela relevante da força de trabalho não teve o regime registrado. Para identificar quais unidades concentram mais planos sem regime, execute a Consulta 2 do mapeamento do Eixo 1 filtrada por `unidade_id`.

### O que fazer quando o resultado estiver vazio

Se a consulta retornar zero linhas, as causas mais prováveis são:

1. **Nenhum plano de trabalho tem `usuario_id` preenchido** — o filtro `pt.usuario_id is not null` elimina planos sem servidor vinculado.
2. **O período de análise está fora da faixa dos dados** — verifique as datas dos planos no banco com a Consulta 3 do mapeamento do Eixo 1.
3. **A tabela `planos_trabalhos` está vazia ou sem registros ativos** — improvável no banco PETRVS via Denodo (14.168 registros confirmados em 15.05.2026), mas verifique com `SELECT COUNT(*) FROM planos_trabalhos WHERE deleted_at IS NULL`.
