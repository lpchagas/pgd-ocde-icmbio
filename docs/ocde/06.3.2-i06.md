# I06 — Grau de Responsabilidade pelas Entregas

## 1. Finalidade do indicador

O I06 calcula **quantos servidores estão vinculados a cada entrega do Plano de Entregas**, e classifica as entregas por tamanho do grupo responsável.

A pergunta central que o I06 responde é: **"As entregas da unidade têm cobertura distribuída entre equipe, ou há entregas que dependem de um único servidor?"**

O resultado traz uma linha por unidade e por faixa de responsáveis, com:

- **`tamanho_grupo_responsavel`** — faixa: "1 servidor", "2 servidores", "3 servidores" ou "4+ servidores"
- **`total_entregas_na_categoria`** — quantas entregas da unidade se enquadram nessa faixa

O indicador **não mede quem são os responsáveis** — isso é o I05. O que ele mede é o nível de compartilhamento do risco: uma entrega com 1 responsável é um ponto único de falha; uma com 4+ servidores tem cobertura distribuída.

**Perspectiva complementar:** o I06 olha o vínculo a partir da entrega ("quantos servidores cada entrega tem?"). O I05 olha o mesmo vínculo a partir do servidor ("quantas entregas cada servidor carrega?"). A leitura dos dois juntos fecha o diagnóstico de distribuição de carga.

## 2. Consulta completa (Denodo/JDBC)

```sql
WITH parametros AS (
    SELECT
        CAST('2025-01-01' AS DATE) AS data_inicio,
        CAST('2025-12-31' AS DATE) AS data_fim,
        0                          AS incluir_excluidos
),
vinculos AS (
    SELECT DISTINCT
        COALESCE(un.sigla, 'N.I.') AS unidade_sigla,
        COALESCE(un.nome,  'N.I.') AS unidade_nome,
        pte.plano_entrega_entrega_id AS id_entrega,
        pt.usuario_id                AS id_servidor
    FROM petrvs_icmbio_planos_trabalhos pt
    JOIN petrvs_icmbio_planos_trabalhos_entregas pte
        ON pte.plano_trabalho_id = pt.id
    LEFT JOIN petrvs_icmbio_unidades un
        ON un.id = pt.unidade_id
    CROSS JOIN parametros p
    WHERE CAST(pt.data_inicio AS DATE) <= p.data_fim
      AND CAST(pt.data_fim   AS DATE) >= p.data_inicio
      AND (p.incluir_excluidos = 1 OR pt.deleted_at  IS NULL)
      AND (p.incluir_excluidos = 1 OR pte.deleted_at IS NULL)
      AND pt.usuario_id IS NOT NULL
      AND pte.plano_entrega_entrega_id IS NOT NULL
),
responsaveis_por_entrega AS (
    SELECT
        unidade_sigla,
        MIN(unidade_nome)           AS unidade_nome,
        id_entrega,
        COUNT(DISTINCT id_servidor) AS qtd_responsaveis
    FROM vinculos
    GROUP BY unidade_sigla, id_entrega
),
com_classificacao AS (
    SELECT
        unidade_sigla,
        unidade_nome,
        id_entrega,
        qtd_responsaveis,
        CASE
            WHEN qtd_responsaveis = 1 THEN '1 servidor'
            WHEN qtd_responsaveis = 2 THEN '2 servidores'
            WHEN qtd_responsaveis = 3 THEN '3 servidores'
            ELSE                           '4+ servidores'
        END AS tamanho_grupo_responsavel
    FROM responsaveis_por_entrega
),
totais_unidade AS (
    SELECT
        unidade_sigla,
        COUNT(id_entrega) AS total_entregas_unidade
    FROM com_classificacao
    GROUP BY unidade_sigla
)
SELECT
    cc.unidade_sigla,
    MIN(cc.unidade_nome)                                               AS unidade_nome,
    cc.tamanho_grupo_responsavel,
    COUNT(cc.id_entrega)                                               AS total_entregas_na_categoria,
    tu.total_entregas_unidade,
    ROUND(COUNT(cc.id_entrega) * 100.0 / NULLIF(tu.total_entregas_unidade, 0), 1)
                                                                       AS pct_categoria
FROM com_classificacao cc
JOIN totais_unidade tu ON tu.unidade_sigla = cc.unidade_sigla
GROUP BY cc.unidade_sigla, cc.tamanho_grupo_responsavel, tu.total_entregas_unidade
ORDER BY cc.unidade_sigla, cc.tamanho_grupo_responsavel
```

**Notas de compatibilidade:**
- Use `CAST('data' AS DATE)` em vez de `DATE('data')` — requerido pelo Denodo VQL
- Prefixo `petrvs_icmbio_` é obrigatório em todos os nomes de tabela (tanto DBeaver quanto JDBC)
- Soft-delete aplicado em ambas as tabelas: `pt.deleted_at IS NULL` e `pte.deleted_at IS NULL`
- Quando executar no DBeaver, remova os prefixos `petrvs_icmbio_` das tabelas

## 3. Passos da consulta

A consulta tem quatro CTEs encadeadas antes do `SELECT` final.

### Passo 1 — Bloco `parametros`: painel de controle

```sql
parametros as (
    select
        date('2025-01-01') as data_inicio,
        date('2025-12-31') as data_fim,
        0 as incluir_excluidos
)
```

Ajuste `data_inicio` e `data_fim` para o período de análise. O filtro é aplicado sobre a **vigência do Plano de Trabalho** — não sobre a data de conclusão de cada entrega.

- `incluir_excluidos = 0` (padrão): apenas planos ativos (`deleted_at is null`)
- `incluir_excluidos = 1`: inclui planos excluídos logicamente — útil para auditoria histórica

Este bloco é idêntico ao do I05. Se as duas consultas forem executadas no mesmo ciclo, use os mesmos valores para garantir comparabilidade.

---

### Passo 2 — Bloco `vinculos`: pares únicos entrega-servidor

```sql
vinculos AS (
    SELECT DISTINCT
        COALESCE(un.sigla, 'N.I.') AS unidade_sigla,
        COALESCE(un.nome,  'N.I.') AS unidade_nome,
        pte.plano_entrega_entrega_id AS id_entrega,
        pt.usuario_id                AS id_servidor
    FROM petrvs_icmbio_planos_trabalhos pt
    JOIN petrvs_icmbio_planos_trabalhos_entregas pte
        ON pte.plano_trabalho_id = pt.id
    LEFT JOIN petrvs_icmbio_unidades un
        ON un.id = pt.unidade_id
    CROSS JOIN parametros p
    WHERE CAST(pt.data_inicio AS DATE) <= p.data_fim
      AND CAST(pt.data_fim   AS DATE) >= p.data_inicio
      AND (p.incluir_excluidos = 1 OR pt.deleted_at  IS NULL)
      AND (p.incluir_excluidos = 1 OR pte.deleted_at IS NULL)
      AND pt.usuario_id IS NOT NULL
      AND pte.plano_entrega_entrega_id IS NOT NULL
)
```

Este bloco é estruturalmente idêntico ao `vinculos_entregas` do I05, com uma diferença de perspectiva: aqui a entidade de interesse é a **entrega** (não o servidor). As colunas retornadas são `id_entrega` e `id_servidor` — o que torna este o bloco neutro entre as duas perspectivas, antes de o próximo bloco fixar o agrupamento.

**Por que `SELECT DISTINCT`:** a tabela `planos_trabalhos_entregas` pode ter múltiplas linhas para o mesmo par (plano de trabalho, entrega) caso o vínculo tenha sido editado. O `DISTINCT` garante que cada par (entrega, servidor) seja contado uma única vez, evitando inflação na contagem de responsáveis.

**Soft-delete em ambas as tabelas:** o filtro `pte.deleted_at IS NULL` foi adicionado na revisão de 24.05.2026, alinhando o I06 com o padrão de correção aplicado ao I05 e I04. Isso garante que registros excluídos logicamente em `planos_trabalhos_entregas` não inflacionem as contagens.

**O `JOIN` interno com `planos_trabalhos_entregas`** garante que apenas planos com entregas vinculadas entrem no resultado. Planos de trabalho cadastrados mas sem nenhuma entrega não aparecem — o que é correto para este indicador.

---

### Passo 3 — Bloco `responsaveis_por_entrega`: contagem de responsáveis por entrega

```sql
responsaveis_por_entrega as (
    select
        unidade_sigla,
        min(unidade_nome)           as unidade_nome,
        id_entrega,
        count(distinct id_servidor) as qtd_responsaveis
    from vinculos
    group by unidade_sigla, id_entrega
)
```

Agrupa os pares (entrega, servidor) por entrega dentro de cada unidade e conta quantos servidores distintos cada entrega tem.

**`count(distinct id_servidor)`:** o `DISTINCT` protege contra o caso em que a mesma entrega de uma unidade apareça vinculada ao mesmo servidor por caminhos diferentes (ex: dois planos de trabalho do mesmo servidor na mesma unidade, ambos vinculados à mesma entrega). Nesse caso, o servidor deve ser contado uma vez, não duas.

**`group by unidade_sigla, id_entrega`:** a entrega é identificada pelo seu `id` na tabela `planos_entregas_entregas`. Uma mesma entrega (mesmo `id`) pertence a uma única unidade — portanto, agrupar por `unidade_sigla, id_entrega` é equivalente a agrupar por `id_entrega`. A `unidade_sigla` é incluída no `GROUP BY` para que ela apareça como coluna no próximo bloco sem necessidade de agregação adicional.

---

### Passo 4 — Bloco `com_classificacao`: faixas de responsabilidade

```sql
com_classificacao as (
    select
        unidade_sigla,
        unidade_nome,
        id_entrega,
        qtd_responsaveis,
        case
            when qtd_responsaveis = 1 then '1 servidor'
            when qtd_responsaveis = 2 then '2 servidores'
            when qtd_responsaveis = 3 then '3 servidores'
            else                           '4+ servidores'
        end as tamanho_grupo_responsavel
    from responsaveis_por_entrega
)
```

Atribui uma das quatro categorias a cada entrega com base no número de responsáveis. As faixas foram definidas para refletir graus crescentes de compartilhamento:

| Faixa | Interpretação de risco |
| --- | --- |
| 1 servidor | Ponto único de falha — se o servidor se afasta, a entrega fica descoberta |
| 2 servidores | Cobertura mínima — há um substituto, mas a saída de qualquer um é crítica |
| 3 servidores | Cobertura razoável — a saída de um não inviabiliza a entrega |
| 4+ servidores | Cobertura ampla — entrega com responsabilidade coletiva distribuída |

A faixa "4+" agrupa todos os valores iguais ou maiores a 4, pois o ganho marginal de segurança entre 4 e 10 responsáveis é decrescente e não justifica categorias separadas para fins de diagnóstico gerencial.

---

### Passo 5 — Bloco `totais_unidade`: totais por unidade (para cálculo de %)

```sql
totais_unidade AS (
    SELECT
        unidade_sigla,
        COUNT(id_entrega) AS total_entregas_unidade
    FROM com_classificacao
    GROUP BY unidade_sigla
)
```

CTE auxiliar que calcula o total de entregas por unidade (soma de todas as faixas). Será reutilizado no `SELECT` final para computar o percentual de cada categoria em relação ao total da unidade.

---

### Passo 6 — SELECT final: contagem de entregas por faixa, unidade e percentual

```sql
SELECT
    cc.unidade_sigla,
    MIN(cc.unidade_nome)                                               AS unidade_nome,
    cc.tamanho_grupo_responsavel,
    COUNT(cc.id_entrega)                                               AS total_entregas_na_categoria,
    tu.total_entregas_unidade,
    ROUND(COUNT(cc.id_entrega) * 100.0 / NULLIF(tu.total_entregas_unidade, 0), 1)
                                                                       AS pct_categoria
FROM com_classificacao cc
JOIN totais_unidade tu ON tu.unidade_sigla = cc.unidade_sigla
GROUP BY cc.unidade_sigla, cc.tamanho_grupo_responsavel, tu.total_entregas_unidade
ORDER BY cc.unidade_sigla, cc.tamanho_grupo_responsavel
```

O `SELECT` final agrega por unidade e por faixa de responsabilidade, e calcula o percentual de entregas em cada faixa em relação ao total de entregas da unidade. A coluna `pct_categoria` permite identificar rapidamente se a unidade tem concentração alta em um tipo de responsabilidade.

**`NULLIF(..., 0)`:** protege contra divisão por zero no caso improvável de uma unidade sem nenhuma entrega vinculada.

**`ROUND(..., 1)`:** redondeia para uma casa decimal, facilitando leitura em relatórios.

## 5. Tabelas da base utilizadas

Principais tabelas usadas neste indicador (via Denodo/JDBC — sempre usar prefixo `petrvs_icmbio_` nas tabelas quando executar via Python/JDBC):

- `petrvs_icmbio_planos_trabalhos` — planos individuais de cada servidor (vigência, identificação do servidor/unidade).
- `petrvs_icmbio_planos_trabalhos_entregas` — vínculos entre planos e entregas (`plano_trabalho_id`, `plano_entrega_entrega_id`); soft-deletable.
- `petrvs_icmbio_planos_entregas_entregas` — metadados da entrega (id, descrição, plano_entrega_id).
- `petrvs_icmbio_unidades` — sigla e nome da unidade.

Colapsa o resultado de uma linha por entrega para uma linha por unidade por faixa. O resultado final mostra quantas entregas de cada unidade se enquadram em cada nível de responsabilidade.

**`min(unidade_nome)`:** como o agrupamento usa `unidade_sigla`, o nome pode ser trazido com `MIN()` — para cada sigla há um único nome correspondente.

**Ordenação por `tamanho_grupo_responsavel`:** os valores `'1 servidor'`, `'2 servidores'`, `'3 servidores'` e `'4+ servidores'` ordenam alfabeticamente de forma que coincide com a ordem lógica de 1 a 4+, o que é conveniente para leitura — mas é uma coincidência da formatação escolhida, não uma propriedade garantida.

## 4. Como interpretar o resultado

### Estrutura do resultado

O resultado final é uma linha **por unidade por faixa de responsabilidade por período**, permitindo análise comparativa de como as entregas são distribuídas ao longo do tempo.

| Coluna | O que informa |
| --- | --- |
| `unidade_sigla` | Sigla da unidade — identificador principal |
| `unidade_nome` | Nome completo da unidade |
| `tamanho_grupo_responsavel` | Faixa: "1 servidor", "2 servidores", "3 servidores" ou "4+ servidores" |
| `total_entregas_na_categoria` | Quantas entregas da unidade se enquadram nessa faixa |
| `total_entregas_unidade` | Total de entregas distintas da unidade (repetido em cada faixa da mesma unidade) |
| `pct_categoria` | Percentual de entregas nessa faixa em relação ao total — facilita identificação de perfil de distribuição |

### Exemplo com dados do ICMBio

Análise do período T1-2025 (01/01/2025 a 31/03/2025):

| unidade_sigla | unidade_nome | tamanho_grupo_responsavel | total_entregas_na_categoria | total_entregas_unidade | pct_categoria |
| --- | --- | --- | --- | --- | --- |
| AUDIT | Auditoria Interna | 1 servidor | 8 | 13 | 61.5 |
| AUDIT | Auditoria Interna | 2 servidores | 4 | 13 | 30.8 |
| AUDIT | Auditoria Interna | 3 servidores | 1 | 13 | 7.7 |
| CGOF | Coord. Geral de Orçamento e Finanças | 1 servidor | 12 | 36 | 33.3 |
| CGOF | Coord. Geral de Orçamento e Finanças | 2 servidores | 18 | 36 | 50.0 |
| CGOF | Coord. Geral de Orçamento e Finanças | 4+ servidores | 6 | 36 | 16.7 |
| CGOV | Coord. de Governança | 1 servidor | 15 | 25 | 60.0 |
| CGOV | Coord. de Governança | 2 servidores | 10 | 25 | 40.0 |
| COGEO | Coord. de Geoprocessamento | 1 servidor | 3 | 9 | 33.3 |
| COGEO | Coord. de Geoprocessamento | 3 servidores | 3 | 9 | 33.3 |
| COGEO | Coord. de Geoprocessamento | 4+ servidores | 3 | 9 | 33.3 |

**Leituras caso a caso:**

**AUDIT — 8 de 13 entregas com apenas 1 responsável (62%):** mais de metade das entregas da unidade são pontos únicos de falha. Para uma unidade de controle interno — onde a perda de continuidade em auditorias em andamento pode comprometer prazos regulatórios — esse perfil é um sinal de gestão de risco relevante. O I05 complementa: quem são os servidores únicos responsáveis? Eles também estão acima da média em quantidade de entregas?

**CGOF — perfil mais distribuído (12 + 18 + 6 = 36 entregas):** a maioria das entregas tem 2 ou mais responsáveis. As 6 com "4+ servidores" provavelmente são entregas de natureza coletiva (ex: processos de prestação de contas que envolvem vários analistas). As 12 com responsável único merecem atenção, especialmente se coincidirem com servidores identificados como sobrecarregados no I05.

**COGEO — distribuição tripartida (3+3+3):** exatamente um terço das entregas em cada uma das faixas 1, 3 e 4+. As 3 entregas com "1 servidor" têm o mesmo peso numérico que as 3 com "4+ servidores". Para uma unidade pequena, isso pode indicar que certas entregas são genuinamente individuais (análises técnicas especializadas) e outras são coletivas (produtos de equipe).

**CGOV — 25 entregas, 15 com responsável único (60%):** concentração alta de pontos únicos de falha na maior unidade do exemplo. Vale investigar se as 15 entregas de responsável único correspondem a servidores especializados (o que pode ser aceitável) ou se é resultado de sub-registro — servidores que deveriam estar vinculados mas não foram cadastrados no plano.

### Nuances importantes

**Responsável único ≠ problema imediato:** algumas entregas são genuinamente individuais por natureza — uma análise jurídica específica, um relatório de responsabilidade de um cargo de confiança, uma consultoria técnica. O I06 identifica a estrutura de responsabilidade, não se ela é adequada. O contexto da entrega determina se "1 servidor" é um risco ou uma característica intrínseca da atividade.

**Entregas sem nenhum servidor vinculado não aparecem no resultado.** O I06 conta apenas entregas que têm ao menos um plano de trabalho vinculado. Entregas registradas no Plano de Entregas mas sem nenhum servidor vinculado em `planos_trabalhos_entregas` são invisíveis aqui — e são um sinal de sub-registro ainda mais grave do que as de responsável único. A consulta de auditoria 1 do [Eixo 3](06.3-eixo3.md) quantifica esse caso.

**Perspectiva crítica da unidade no I06 vs. I02/I03/I04:** a unidade no I06 vem de `pt.unidade_id` (unidade do **executor**, que é o servidor), não de `pe.unidade_id` (unidade que elaborou o PE). Isso significa:

- Uma entrega planejada pela CGOV mas executada por servidor da AUDIT aparecerá sob AUDIT no I06, não sob CGOV
- Uma mesma entrega pode aparecer em múltiplas unidades no resultado do I06 se executores de diferentes unidades a tiverem no plano
- Isso explica por que `total_i06 > total_pe_por_unidade` é possível — entregas cruzadas se distribuem entre as unidades dos executores
- Em comparações, use `total_entregas_unidade` (coluna presente no CSV) como base, não totais de PE

Esta é uma característica intencional: o I06 mede responsabilidade de quem executa, não de quem planejou. Para diagnóstico de sobrecarga, isso é mais relevante.

### Leitura integrada com outros indicadores

| Situação observada no I06 | Próximo indicador | Pergunta a responder |
| --- | --- | --- |
| Muitas entregas com "1 servidor" | I05 | Os responsáveis únicos são os mesmos servidores sobrecarregados? |
| Poucas entregas com "4+ servidores" | I02 | As entregas coletivas têm taxa de cumprimento maior do que as individuais? |
| Unidade com perfil concentrado em "1 servidor" | I07 | Quantas horas representam essas entregas individuais? |
| Comparação entre ciclos | — | A proporção de entregas com responsável único está crescendo ou reduzindo ao longo dos ciclos? |

## 7. Avisos metodológicos e limitações

### Ciclos mistos — não compare totais entre períodos de anos diferentes

O ICMBio adota:
- 2025: **trimestral** (T1: Jan–Mar | T2: Abr–Jun | T3: Jul–Set | T4: Out–Dez)
- 2026+: **quadrimestral** (Q1: Jan–Abr | Q2: Mai–Ago | Q3: Set–Dez)

Que um período tenha 90 dias (trimestre) e outro 120 dias (quadrimestre) afeta o número de servidores com PT ativo — portanto, não é válido comparar `total_entregas_unidade` de T1-2025 com Q1-2026 diretamente. Use apenas `pct_categoria` para comparação de estrutura de responsabilidade entre períodos de comprimentos diferentes.

### O filtro temporal incide sobre a vigência do PT

O indicador filtra por `CAST(pt.data_inicio AS DATE) <= data_fim AND CAST(pt.data_fim AS DATE) >= data_inicio` — ou seja, retorna apenas servidores cujo Plano de Trabalho se sobrepõe ao período analisado. Um servidor cujo PT vai de maio a junho não aparecerá em T1-2025 (jan–mar), mesmo que tenha entregas cadastradas para aquele período no PE.

Conseguência: contagens podem ser menores que as exibidas no PETRVS, que mostra todas as entregas históricas sem recorte temporal.

### Entregas sem servidor vinculado são invisíveis

O I06 conta apenas entregas que têm ao menos um servidor vinculado em `planos_trabalhos_entregas`. Entregas do PE sem nenhum vínculo em PT ativo não aparecem — o que é um sinal de sub-registro ainda mais grave do que responsável único.

Para quantificar essa invisibilidade, compare `total_entregas_unidade` do I06 com `total_entregas_no_ciclo` do I04/I05 por unidade.

### Avisos no CSV

O script Python que gera o CSV (`IND_06.1_run.py`) exibe avisos ao final:
- Reafirma que "unidade = unidade do SERVIDOR"
- Alerta sobre possibilidade de `total_i06 ≠ total_pe`
- Avisa que ciclos mistos exigem cuidado em comparações
