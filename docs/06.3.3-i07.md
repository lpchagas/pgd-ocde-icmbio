# I07 — Horas por Entrega — Planejadas (Absolutas)

## 1. Finalidade do indicador

O I07 calcula **o total de horas de trabalho planejadas alocadas a cada entrega do Plano de Entregas**, somando a contribuição de todos os servidores cujos Planos de Trabalho cobrem o período e que tenham aquela entrega vinculada.

A pergunta central que o I07 responde é: **"Quanto esforço humano foi planejado para cada entrega — em horas absolutas?"**

O resultado traz uma linha por entrega com:

- **`total_horas_planejadas_entrega`** — soma das horas de todos os servidores alocados à entrega, ponderada pelo campo `forca_trabalho` de cada vínculo

O I07 trabalha com **horas planejadas, não realizadas**. O PETRVS registra a intenção de alocação no momento do cadastro do Plano de Trabalho — não o tempo efetivamente dedicado a cada entrega após o ciclo. Esse é um diagnóstico de planejamento, não de execução.

**Perspectiva complementar:** o I07 mostra o valor absoluto em horas. O I08 mostra o mesmo valor como percentual da capacidade total da unidade. Para comparar o peso relativo de entregas entre unidades de tamanhos diferentes, use o I08. Para dimensionar o esforço em termos operacionais ("quantas semanas de trabalho representa essa entrega?"), use o I07.

## 2. Consulta completa (MySQL)

```sql
with recursive
parametros as (
    select
        date('2025-01-01') as data_inicio,
        date('2025-12-31') as data_fim,
        8                  as horas_por_dia,
        0                  as incluir_excluidos
),
anos as (
    select year((select data_inicio from parametros)) as ano
    union all
    select ano + 1
    from anos
    where ano < year((select data_fim from parametros))
),
feriados_fixos as (
    select date(concat(ano, '-01-01')) as data_feriado from anos
    union all select date(concat(ano, '-04-21')) from anos
    union all select date(concat(ano, '-05-01')) from anos
    union all select date(concat(ano, '-09-07')) from anos
    union all select date(concat(ano, '-10-12')) from anos
    union all select date(concat(ano, '-11-02')) from anos
    union all select date(concat(ano, '-11-15')) from anos
    union all select date(concat(ano, '-11-20')) from anos
    union all select date(concat(ano, '-12-25')) from anos
),
feriados_moveis as (
    select null as data_feriado where 1 = 0
    -- union all select date('2025-04-18')  -- Sexta-feira da Paixão 2025
    -- union all select date('2026-04-03')  -- Sexta-feira da Paixão 2026
),
feriados_nacionais as (
    select data_feriado from feriados_fixos
    union
    select data_feriado from feriados_moveis
),
calendario as (
    select (select data_inicio from parametros) as data_dia
    union all
    select date_add(data_dia, interval 1 day)
    from calendario
    where data_dia < (select data_fim from parametros)
),
links_distintos as (
    select
        pte.plano_trabalho_id,
        pte.plano_entrega_entrega_id   as id_entrega,
        coalesce(pte.forca_trabalho, 0) as forca_trabalho
    from planos_trabalhos_entregas pte
    where pte.plano_entrega_entrega_id is not null
),
horas_planejadas_por_plano as (
    select
        pt.id          as plano_trabalho_id,
        pt.unidade_id,
        count(c.data_dia)                          as dias_uteis_plano,
        count(c.data_dia) * p.horas_por_dia        as horas_planejadas_plano
    from planos_trabalhos pt
    cross join parametros p
    join calendario c
        on c.data_dia between greatest(date(pt.data_inicio), p.data_inicio)
                          and least(date(pt.data_fim),   p.data_fim)
    left join feriados_nacionais fn
        on fn.data_feriado = c.data_dia
    where date(pt.data_inicio) <= p.data_fim
      and date(pt.data_fim)   >= p.data_inicio
      and (p.incluir_excluidos = 1 or pt.deleted_at is null)
      and dayofweek(c.data_dia) not in (1, 7)
      and fn.data_feriado is null
    group by pt.id, pt.unidade_id, p.horas_por_dia
),
horas_alocadas_por_entrega as (
    select
        coalesce(un.sigla, 'N.I.') as unidade_sigla,
        coalesce(un.nome, 'N.I.')  as unidade_nome,
        ld.id_entrega,
        coalesce(
            nullif(trim(coalesce(pee.descricao, '')), ''),
            nullif(trim(coalesce(pee.descricao_entrega, '')), ''),
            'N.I.'
        )                          as nome_entrega,
        pe.id                      as id_plano_entrega,
        date(pe.data_inicio)       as inicio_vigencia_plano_entrega,
        date(pe.data_fim)          as fim_vigencia_plano_entrega,
        hpp.horas_planejadas_plano * (ld.forca_trabalho / 100.0) as horas_planejadas_alocadas
    from links_distintos ld
    join horas_planejadas_por_plano hpp
        on hpp.plano_trabalho_id = ld.plano_trabalho_id
    left join planos_entregas_entregas pee
        on pee.id = ld.id_entrega
    left join planos_entregas pe
        on pe.id = pee.plano_entrega_id
    left join unidades un
        on un.id = hpp.unidade_id
)
select
    unidade_sigla,
    unidade_nome,
    id_entrega,
    nome_entrega,
    id_plano_entrega,
    inicio_vigencia_plano_entrega,
    fim_vigencia_plano_entrega,
    round(sum(horas_planejadas_alocadas), 2) as total_horas_planejadas_entrega
from horas_alocadas_por_entrega
group by unidade_sigla, unidade_nome, id_entrega, nome_entrega,
         id_plano_entrega, inicio_vigencia_plano_entrega, fim_vigencia_plano_entrega
order by unidade_sigla, total_horas_planejadas_entrega desc;
```

> **Requisito técnico:** esta consulta usa `WITH RECURSIVE` para geração de calendário e requer **MySQL 8.0 ou superior**. O limite padrão de recursão é 1.000 iterações (~2,7 anos). Para períodos maiores, execute `SET SESSION cte_max_recursion_depth = 5000;` antes da query.

## 3. Passos da consulta

A consulta tem oito CTEs encadeadas antes do `SELECT` final. Os quatro primeiros constroem o calendário de dias úteis; os quatro seguintes calculam as horas por entrega.

### Passo 1 — Bloco `parametros`: painel de controle

```sql
parametros as (
    select
        date('2025-01-01') as data_inicio,
        date('2025-12-31') as data_fim,
        8                  as horas_por_dia,
        0                  as incluir_excluidos
)
```

Este bloco tem um campo extra em relação aos indicadores anteriores: `horas_por_dia`. O padrão é `8`, correspondendo à jornada federal padrão de 8 horas diárias. Se a organização ou a unidade operar com jornada diferente (ex: 6 horas para cargos específicos), ajuste este valor.

`data_inicio` e `data_fim` delimitam tanto o período de análise quanto a janela do calendário de dias úteis gerado. Todos os planos de trabalho com vigência sobreposta a esse intervalo entram no cálculo.

---

### Passo 2 — Blocos `anos`, `feriados_fixos`, `feriados_moveis`, `feriados_nacionais`: calendário de feriados

```sql
anos as (
    select year((select data_inicio from parametros)) as ano
    union all
    select ano + 1 from anos
    where ano < year((select data_fim from parametros))
),
feriados_fixos as (
    select date(concat(ano, '-01-01')) as data_feriado from anos
    union all select date(concat(ano, '-04-21')) from anos
    -- ... (demais feriados fixos nacionais)
),
feriados_moveis as (
    select null as data_feriado where 1 = 0
    -- union all select date('2025-04-18')  -- descomente e adicione datas reais
),
feriados_nacionais as (
    select data_feriado from feriados_fixos
    union
    select data_feriado from feriados_moveis
)
```

**Por que quatro blocos para feriados:**

O bloco `anos` gera a lista de anos cobertos pelo período (ex: 2025 para um análise anual; 2025 e 2026 para um período que atravessa a virada de ano). Isso permite que `feriados_fixos` replique as datas fixas para cada ano do período automaticamente, sem precisar listar manualmente cada ano.

Os 9 feriados nacionais fixos gerados são:

| Data | Feriado |
| --- | --- |
| 01/01 | Confraternização Universal |
| 21/04 | Tiradentes |
| 01/05 | Dia do Trabalho |
| 07/09 | Independência do Brasil |
| 12/10 | Nossa Senhora Aparecida |
| 02/11 | Finados |
| 15/11 | Proclamação da República |
| 20/11 | Consciência Negra |
| 25/12 | Natal |

O bloco `feriados_moveis` está presente mas vazio por padrão (`select null where 1 = 0` não retorna linhas). Para incluir Sexta-feira da Paixão e Corpus Christi, descomente as linhas correspondentes e insira as datas corretas para os anos do período:

```sql
feriados_moveis as (
    select null as data_feriado where 1 = 0
    union all select date('2025-04-18')  -- Sexta-feira da Paixão 2025
    union all select date('2025-06-19')  -- Corpus Christi 2025
    union all select date('2026-04-03')  -- Sexta-feira da Paixão 2026
    union all select date('2026-06-04')  -- Corpus Christi 2026
)
```

O `UNION` (sem `ALL`) em `feriados_nacionais` elimina duplicatas entre as duas listas, evitando que um feriado seja removido duas vezes do calendário.

---

### Passo 3 — Bloco `calendario`: sequência de dias úteis via CTE recursiva

```sql
calendario as (
    select (select data_inicio from parametros) as data_dia
    union all
    select date_add(data_dia, interval 1 day)
    from calendario
    where data_dia < (select data_fim from parametros)
)
```

Esta é a CTE recursiva central da consulta. Ela gera uma linha para **cada dia do calendário** entre `data_inicio` e `data_fim` — incluindo fins de semana e feriados. A filtragem dos dias não úteis acontece no próximo bloco.

**Como a recursão funciona:**

1. A linha âncora (`select data_inicio`) inicia com o primeiro dia do período.
2. A parte recursiva (`select date_add(data_dia, interval 1 day)`) adiciona um dia a cada iteração.
3. A condição de parada (`where data_dia < data_fim`) encerra quando o último dia é gerado.

**Limite de recursão:** o MySQL permite por padrão até 1.000 iterações em uma CTE recursiva. Para 365 dias (1 ano), o limite é suficiente. Para períodos maiores (ex: 3 anos = ~1.095 dias), execute antes da query:

```sql
set session cte_max_recursion_depth = 5000;
```

---

### Passo 4 — Bloco `links_distintos`: vínculos plano-entrega com força de trabalho

```sql
links_distintos as (
    select
        pte.plano_trabalho_id,
        pte.plano_entrega_entrega_id   as id_entrega,
        coalesce(pte.forca_trabalho, 0) as forca_trabalho
    from planos_trabalhos_entregas pte
    where pte.plano_entrega_entrega_id is not null
)
```

Extrai os vínculos entre planos de trabalho e entregas, incluindo o campo `forca_trabalho`. O `coalesce(pte.forca_trabalho, 0)` trata valores nulos como zero — um servidor vinculado a uma entrega sem `forca_trabalho` preenchido contribui com zero horas para ela.

Este bloco não aplica filtro de período — o filtro é aplicado no próximo bloco, onde os planos de trabalho são cruzados com o calendário. Isso garante que a alocação seja calculada apenas para os dias do período efetivamente cobertos pelo plano.

---

### Passo 5 — Bloco `horas_planejadas_por_plano`: horas úteis de cada plano no período

```sql
horas_planejadas_por_plano as (
    select
        pt.id          as plano_trabalho_id,
        pt.unidade_id,
        count(c.data_dia)                   as dias_uteis_plano,
        count(c.data_dia) * p.horas_por_dia as horas_planejadas_plano
    from planos_trabalhos pt
    cross join parametros p
    join calendario c
        on c.data_dia between greatest(date(pt.data_inicio), p.data_inicio)
                          and least(date(pt.data_fim),   p.data_fim)
    left join feriados_nacionais fn
        on fn.data_feriado = c.data_dia
    where date(pt.data_inicio) <= p.data_fim
      and date(pt.data_fim)   >= p.data_inicio
      and (p.incluir_excluidos = 1 or pt.deleted_at is null)
      and dayofweek(c.data_dia) not in (1, 7)
      and fn.data_feriado is null
    group by pt.id, pt.unidade_id, p.horas_por_dia
)
```

Este é o bloco que converte vigências de planos em horas. Para cada plano de trabalho, ele percorre o calendário e conta apenas os dias úteis que caem dentro da interseção entre a vigência do plano e o período de análise.

**A interseção de períodos:**

```sql
c.data_dia between greatest(date(pt.data_inicio), p.data_inicio)
              and least(date(pt.data_fim), p.data_fim)
```

Um plano de trabalho com vigência de outubro a março, analisado num período de janeiro a dezembro, terá dias contados apenas de janeiro a março — a parte da vigência que cai dentro do período de análise. `GREATEST` escolhe o início mais tardio; `LEAST` escolhe o fim mais cedo.

**Exclusão de fins de semana:** `dayofweek(c.data_dia) not in (1, 7)` — no MySQL, `1 = Domingo` e `7 = Sábado`.

**Exclusão de feriados:** o `LEFT JOIN` com `feriados_nacionais` combina cada dia do calendário com a tabela de feriados. O filtro `fn.data_feriado is null` seleciona apenas os dias em que não houve correspondência — ou seja, os dias que não são feriados.

**O resultado:** cada plano de trabalho produz uma linha com `dias_uteis_plano` e `horas_planejadas_plano = dias_uteis_plano × horas_por_dia`.

---

### Passo 6 — Bloco `horas_alocadas_por_entrega`: distribuição de horas por entrega

```sql
horas_alocadas_por_entrega as (
    select
        ...
        hpp.horas_planejadas_plano * (ld.forca_trabalho / 100.0) as horas_planejadas_alocadas
    from links_distintos ld
    join horas_planejadas_por_plano hpp on hpp.plano_trabalho_id = ld.plano_trabalho_id
    left join planos_entregas_entregas pee on pee.id = ld.id_entrega
    left join planos_entregas pe on pe.id = pee.plano_entrega_id
    left join unidades un on un.id = hpp.unidade_id
)
```

Aplica a fórmula central do I07:

```
horas_alocadas = horas_totais_do_plano × (forca_trabalho / 100)
```

**Exemplo numérico:** um plano com 200 dias úteis no período (1.600 horas a 8h/dia) e `forca_trabalho = 30` para uma entrega contribui com `1.600 × 0,30 = 480 horas` para essa entrega.

O `JOIN` interno com `horas_planejadas_por_plano` garante que apenas planos com dias úteis calculados entrem no resultado. O `LEFT JOIN` com as demais tabelas preserva vínculos cujos metadados de entrega (nome, vigência do PE) estejam incompletos — eles aparecem com `'N.I.'` em vez de serem descartados.

---

### Passo 7 — SELECT final: total de horas por entrega

```sql
select
    unidade_sigla,
    unidade_nome,
    id_entrega,
    nome_entrega,
    id_plano_entrega,
    inicio_vigencia_plano_entrega,
    fim_vigencia_plano_entrega,
    round(sum(horas_planejadas_alocadas), 2) as total_horas_planejadas_entrega
from horas_alocadas_por_entrega
group by unidade_sigla, unidade_nome, id_entrega, nome_entrega,
         id_plano_entrega, inicio_vigencia_plano_entrega, fim_vigencia_plano_entrega
order by unidade_sigla, total_horas_planejadas_entrega desc;
```

Agrupa todas as contribuições individuais por entrega e soma as horas. O `SUM` consolida a contribuição de múltiplos servidores numa mesma entrega.

O resultado é ordenado por unidade e por horas decrescentes — as entregas que mais absorvem capacidade aparecem primeiro dentro de cada unidade.

As colunas `id_plano_entrega`, `inicio_vigencia_plano_entrega` e `fim_vigencia_plano_entrega` fornecem contexto sobre o ciclo de planejamento ao qual a entrega pertence — úteis para identificar se a entrega está em um PE ativo ou encerrado.

## 4. Como interpretar o resultado

### Estrutura do resultado

| Coluna | O que informa |
| --- | --- |
| `unidade_sigla` | Sigla da unidade |
| `unidade_nome` | Nome completo da unidade |
| `id_entrega` | Identificador da entrega em `planos_entregas_entregas` |
| `nome_entrega` | Nome da entrega (campo `descricao` com fallback para `descricao_entrega`) |
| `id_plano_entrega` | Identificador do Plano de Entregas ao qual a entrega pertence |
| `inicio_vigencia_plano_entrega` | Data de início do PE |
| `fim_vigencia_plano_entrega` | Data de fim do PE |
| `total_horas_planejadas_entrega` | Soma das horas planejadas de todos os servidores alocados à entrega, ponderada por `forca_trabalho` |

### Exemplo com dados do ICMBio

Análise do período 2025-01-01 a 2025-12-31. Entregasordenadas por horas:

| unidade_sigla | nome_entrega | id_plano_entrega | inicio_vigencia | fim_vigencia | total_horas_planejadas_entrega |
| --- | --- | --- | --- | --- | --- |
| CGOV | Relatório anual de dimensionamento | 12 | 2025-01-01 | 2025-12-31 | 1.280,00 |
| CGOV | Mapeamento de processos críticos | 12 | 2025-01-01 | 2025-12-31 | 960,00 |
| CGOV | Nota técnica de avaliação institucional | 12 | 2025-01-01 | 2025-12-31 | 320,00 |
| COGEO | Atualização do banco geoespacial | 15 | 2025-01-01 | 2025-06-30 | 2.400,00 |
| COGEO | Modelagem de distribuição de espécies | 15 | 2025-01-01 | 2025-06-30 | 800,00 |
| AUDIT | Auditoria de contratos externos | 18 | 2025-04-01 | 2025-12-31 | 480,00 |
| AUDIT | Revisão de processos de licitação | 18 | 2025-04-01 | 2025-12-31 | 160,00 |

**Leituras caso a caso:**

**COGEO — "Atualização do banco geoespacial" (2.400 horas):** esta é a entrega com maior carga absoluta do exemplo. 2.400 horas equivalem a aproximadamente 300 dias de trabalho de 8h — o equivalente a um servidor dedicado exclusivamente a essa entrega por um ano e dois meses. Para um PE semestral (jan–jun), isso indica que múltiplos servidores estão alocados, cada um com alta `forca_trabalho` nessa entrega. O I06 revela quantos são; o I08 mostrará qual percentual da capacidade da COGEO essa entrega representa.

**CGOV — distribuição entre três entregas:** as três entregas somam 2.560 horas. A entrega com maior peso ("Relatório anual", 1.280h) absorve 50% do total das três. O I08 calculará quanto isso representa da capacidade total da CGOV no período.

**AUDIT — "Auditoria de contratos externos" (480 horas) vs "Revisão de processos" (160 horas):** proporção de 3:1. A entrega mais pesada absorve três vezes mais esforço que a mais leve — informação relevante para priorização de acompanhamento e alocação de recursos.

**Referência rápida de conversão:**

| Total de horas | Equivalente aproximado |
| --- | --- |
| 8 h | 1 dia de trabalho |
| 40 h | 1 semana de trabalho |
| 160 h | 1 mês de trabalho (20 dias úteis) |
| 1.920 h | 1 ano de trabalho de um servidor (240 dias úteis) |

### Nuances importantes

**Horas acima da capacidade de um servidor são possíveis.** Se uma entrega tem 4 servidores, cada um com `forca_trabalho = 50`, o total pode facilmente superar 1.000 horas. Isso não é um erro — é a soma da contribuição coletiva planejada.

**O resultado pode incluir entregas de PEs encerrados.** O filtro é sobre a vigência do **Plano de Trabalho**, não do PE. Um servidor com plano ativo em 2025 que tinha vínculo com uma entrega de um PE de 2024 pode gerar linha no resultado. As colunas `inicio_vigencia_plano_entrega` e `fim_vigencia_plano_entrega` permitem identificar esse caso.

**`total_horas_planejadas_entrega = 0` indica `forca_trabalho = 0` em todos os vínculos.** A entrega tem servidores associados (aparece no I05 e I06), mas nenhum declarou percentual de alocação. Vale retornar ao cadastro dos planos e verificar se o campo foi preenchido.

### Leitura integrada com outros indicadores

| Situação observada no I07 | Próximo indicador | Pergunta a responder |
| --- | --- | --- |
| Entrega com horas muito acima das demais | I08 | Qual percentual da capacidade da unidade essa entrega representa? |
| Entrega com 0 horas mas com responsáveis (I06) | — | O campo `forca_trabalho` está preenchido nos planos? (consulta de auditoria) |
| Entrega crítica com poucas horas planejadas | I02 | Essa entrega foi cumprida apesar da baixa alocação? |
| Comparação entre entregas do mesmo PE | — | A distribuição de horas é proporcional à complexidade das entregas? |
