# I10 — Percentual de Avaliações Inadequadas

## i. Finalidade do indicador

O I10 calcula a **proporção de avaliações de Planos de Trabalho que receberam nota 2 ("Inadequado")** em relação ao total de avaliações de PT da unidade no período.

Se o I09 dá uma visão geral da qualidade média das avaliações, o I10 olha especificamente para o **extremo negativo da distribuição**: qual é o peso das avaliações que identificaram desempenho formalmente abaixo do mínimo aceitável?

A pergunta central que o I10 responde é: **"Qual percentual dos Planos de Trabalho avaliados no período foram classificados como 'Inadequado'?"**

O indicador foca exclusivamente na nota 2 — e não na nota 1 ("Não executado") nem em uma faixa ampla. Essa escolha segue a especificação do *Performance Toolkit* da OCDE: a nota "Inadequado" representa o padrão de alerta operacional mais relevante, porque documenta um servidor que atuou, teve o trabalho avaliado e foi formalmente reconhecido como aquém do mínimo — diferente da nota 1, que tipicamente indica ausência de execução.

O resultado é uma linha por unidade com o total de avaliações, a contagem e o percentual de avaliações com nota 2 e um nível de alerta categorizado. Esse conjunto permite:

- Identificar unidades com concentração preocupante de desempenho "Inadequado";
- Monitorar a evolução do indicador ao longo dos ciclos (tendência de redução ou aumento);
- Priorizar unidades para ações de desenvolvimento e suporte de gestão;
- Cruzar com o I12 para verificar se a unidade reconhece coletivamente (PE) o mesmo padrão visto nas avaliações individuais (PT).

> **Pré-requisito:** antes de executar esta consulta, execute as consultas de mapeamento do [Eixo 4](06.4-eixo4.md) para confirmar que `nota = 2` corresponde à categoria "Inadequado" no seu banco.

## ii. Consulta completa (MySQL)

```sql
with parametros as (
    select
        date('2025-01-01') as data_inicio,
        date('2025-12-31') as data_fim,
        0                  as incluir_excluidos
),
avaliacoes_pt as (
    select
        av.id          as id_avaliacao,
        pt.unidade_id,
        tan.nota       as valor_nota
    from avaliacoes av
    join planos_trabalhos_consolidacoes ptc
        on ptc.id = av.plano_trabalho_consolidacao_id
    join planos_trabalhos pt
        on pt.id = ptc.plano_trabalho_id
    join tipos_avaliacoes_notas tan
        on tan.id = av.tipo_avaliacao_nota_id
    cross join parametros p
    where av.plano_trabalho_consolidacao_id is not null
      and (p.incluir_excluidos = 1 or av.deleted_at is null)
      and date(pt.data_inicio) <= p.data_fim
      and date(pt.data_fim)   >= p.data_inicio
      and (p.incluir_excluidos = 1 or pt.deleted_at is null)
),
proporcao_por_unidade as (
    select
        coalesce(un.sigla, 'N.I.')                                   as unidade_sigla,
        coalesce(un.nome, 'N.I.')                                    as unidade_nome,
        count(avpt.id_avaliacao)                                     as total_avaliacoes_pt,
        sum(case when avpt.valor_nota = 2 then 1 else 0 end)         as qtd_inadequado,
        round(
            sum(case when avpt.valor_nota = 2 then 1 else 0 end)
                / nullif(count(avpt.id_avaliacao), 0) * 100,
            2
        )                                                            as perc_inadequado
    from avaliacoes_pt avpt
    left join unidades un
        on un.id = avpt.unidade_id
    group by coalesce(un.sigla, 'N.I.'), coalesce(un.nome, 'N.I.')
)
select
    unidade_sigla,
    unidade_nome,
    total_avaliacoes_pt,
    qtd_inadequado,
    perc_inadequado,
    case
        when perc_inadequado >= 30 then 'Atenção crítica'
        when perc_inadequado >= 15 then 'Atenção moderada'
        when perc_inadequado >= 5  then 'Observação'
        else 'Baixa prevalência'
    end as nivel_alerta
from proporcao_por_unidade
order by perc_inadequado desc, unidade_sigla;
```

## iii. Passos da consulta

A consulta segue a mesma arquitetura em três CTEs do I09. Os dois primeiros blocos são idênticos — a diferença está no terceiro bloco e no `SELECT` final, onde o foco se estreita para a nota 2.

### Passo 1 — Bloco `parametros`: painel de controle da consulta

```sql
parametros as (
    select
        date('2025-01-01') as data_inicio,
        date('2025-12-31') as data_fim,
        0                  as incluir_excluidos
)
```

Idêntico ao I09. Define o período e o comportamento para registros excluídos logicamente. Ajuste apenas `data_inicio` e `data_fim` para o período desejado — o restante da consulta se adapta automaticamente.

Para comparar o I10 entre ciclos (ex: 2024 vs. 2025), execute a consulta duas vezes com períodos diferentes e compare os resultados lado a lado.

---

### Passo 2 — Bloco `avaliacoes_pt`: coletando as avaliações de Plano de Trabalho

```sql
avaliacoes_pt as (
    select
        av.id          as id_avaliacao,
        pt.unidade_id,
        tan.nota       as valor_nota
    from avaliacoes av
    join planos_trabalhos_consolidacoes ptc
        on ptc.id = av.plano_trabalho_consolidacao_id
    join planos_trabalhos pt
        on pt.id = ptc.plano_trabalho_id
    join tipos_avaliacoes_notas tan
        on tan.id = av.tipo_avaliacao_nota_id
    cross join parametros p
    where av.plano_trabalho_consolidacao_id is not null
      and (p.incluir_excluidos = 1 or av.deleted_at is null)
      and date(pt.data_inicio) <= p.data_fim
      and date(pt.data_fim)   >= p.data_inicio
      and (p.incluir_excluidos = 1 or pt.deleted_at is null)
)
```

Idêntico ao I09. Percorre a cadeia `avaliacoes → planos_trabalhos_consolidacoes → planos_trabalhos → unidade_id` para identificar, de cada avaliação, qual unidade está sendo avaliada e qual foi a nota numérica registrada.

**Nota importante:** este bloco coleta **todas** as avaliações de PT do período — incluindo notas 1, 3, 4 e 5. O filtro para nota 2 é aplicado apenas no próximo passo, no momento do cálculo. Isso é necessário para que o denominador (total de avaliações) seja correto.

---

### Passo 3 — Bloco `proporcao_por_unidade`: calculando o percentual de inadequados

Este é o bloco que diferencia o I10 do I09. Em vez de calcular a média de todas as notas, ele conta especificamente quantas avaliações receberam nota 2 e divide pelo total.

```sql
proporcao_por_unidade as (
    select
        coalesce(un.sigla, 'N.I.')                                   as unidade_sigla,
        coalesce(un.nome, 'N.I.')                                    as unidade_nome,
        count(avpt.id_avaliacao)                                     as total_avaliacoes_pt,
        sum(case when avpt.valor_nota = 2 then 1 else 0 end)         as qtd_inadequado,
        round(
            sum(case when avpt.valor_nota = 2 then 1 else 0 end)
                / nullif(count(avpt.id_avaliacao), 0) * 100,
            2
        )                                                            as perc_inadequado
    from avaliacoes_pt avpt
    left join unidades un
        on un.id = avpt.unidade_id
    group by coalesce(un.sigla, 'N.I.'), coalesce(un.nome, 'N.I.')
)
```

**O numerador — `sum(case when avpt.valor_nota = 2 then 1 else 0 end)`:**

Este padrão de `case when` dentro de `sum` é a forma SQL de contar registros que atendem a uma condição específica sem usar uma subconsulta. Para cada avaliação, o `case` verifica: "a nota é 2?". Se sim, contribui com 1; se não, com 0. A soma de todos esses valores é a contagem de avaliações "Inadequado".

O `valor_nota = 2` é um filtro pontual — exatamente a nota 2, sem incluir a nota 1. Isso é deliberado: o I10 mede especificamente o nível "Inadequado" da escala, não um intervalo de baixo desempenho. A nota 1 ("Não executado") representa uma situação mais grave e distinta, e deve ser monitorada separadamente se necessário.

**O denominador — `nullif(count(avpt.id_avaliacao), 0)`:**

Conta o total de avaliações de PT da unidade, independentemente da nota recebida. O `nullif(..., 0)` é uma proteção: se por algum motivo uma unidade não tiver avaliações (o que não deveria acontecer neste ponto, pois o bloco anterior só retorna avaliações existentes), a divisão retornaria `NULL` em vez de erro.

**A fórmula completa:**

```
perc_inadequado = (qtd_inadequado / total_avaliacoes_pt) * 100
```

Se uma unidade tem 40 avaliações de PT e 6 delas têm nota 2, o `perc_inadequado` será `(6 / 40) * 100 = 15.00%`.

---

### Passo 4 — SELECT final: classificando o nível de alerta

```sql
select
    unidade_sigla,
    unidade_nome,
    total_avaliacoes_pt,
    qtd_inadequado,
    perc_inadequado,
    case
        when perc_inadequado >= 30 then 'Atenção crítica'
        when perc_inadequado >= 15 then 'Atenção moderada'
        when perc_inadequado >= 5  then 'Observação'
        else 'Baixa prevalência'
    end as nivel_alerta
from proporcao_por_unidade
order by perc_inadequado desc, unidade_sigla;
```

**Os limiares de alerta:**

| Nível | Condição | Interpretação |
| --- | --- | --- |
| Atenção crítica | `>= 30%` | 1 em cada 3 ou mais avaliações é "Inadequado" — padrão sistêmico, não pontual |
| Atenção moderada | `>= 15%` | Entre 1 em 7 e 1 em 3 — presença relevante que merece aprofundamento |
| Observação | `>= 5%` | Ocorrências isoladas — monitorar para verificar se é tendência ou situação pontual |
| Baixa prevalência | `< 5%` | Menos de 1 em 20 avaliações — dentro do esperado para qualquer conjunto |

Esses limiares são parâmetros de referência e podem ser ajustados conforme o contexto e a maturidade avaliativa da organização. No início da implantação do PGD, quando a cultura de avaliação ainda está sendo construída, limiares mais flexíveis podem ser adequados. Em ciclos maduros, o critério de alerta pode ser mais exigente.

**A ordenação `order by perc_inadequado desc`:** apresenta primeiro as unidades com maior proporção de avaliações inadequadas — as que demandam atenção prioritária ficam no topo.

## iv. Como interpretar o resultado

### Estrutura do resultado

Cada linha representa uma unidade com Planos de Trabalho avaliados no período:

| Coluna | O que informa |
| --- | --- |
| `unidade_sigla` / `unidade_nome` | Identidade da unidade |
| `total_avaliacoes_pt` | Base total de avaliações de PT — o denominador |
| `qtd_inadequado` | Número absoluto de avaliações com nota 2 |
| `perc_inadequado` | Percentual do total — o indicador central |
| `nivel_alerta` | Classificação qualitativa do nível de preocupação |

### Exemplo com dados do ICMBio

Resultado hipotético de uma análise de 2025 (mesmas unidades do exemplo do I09):

| unidade_sigla | total_avaliacoes_pt | qtd_inadequado | perc_inadequado | nivel_alerta |
| --- | --- | --- | --- | --- |
| DIREC | 12 | 4 | 33.33 | Atenção crítica |
| CGOV | 45 | 4 | 8.89 | Observação |
| COGEO | 38 | 2 | 5.26 | Observação |
| CGOF | 52 | 3 | 5.77 | Observação |
| AUDIT | 18 | 0 | 0.00 | Baixa prevalência |

**Leitura caso a caso:**

**DIREC — Atenção crítica (33.33%):** 4 das 12 avaliações de PT foram classificadas como "Inadequado". Este resultado, combinado com a média 2.42 vista no I09, traça um quadro coerente: a unidade tem padrão de desempenho reconhecidamente abaixo do esperado pelos próprios atores do sistema. Requer aprofundamento de causa: as dificuldades são de capacidade técnica, de dimensionamento de equipe, de processos internos ou de definição de metas irrealistas?

**CGOV, COGEO, CGOF — Observação (5-9%):** percentuais no limiar inferior da faixa de observação. Isoladamente, não representam alerta grave, mas devem ser monitorados nos próximos ciclos. Uma tendência de crescimento de um ciclo para o outro é mais preocupante que o valor absoluto atual.

**AUDIT — Baixa prevalência (0%):** zero avaliações com nota 2. Combinado com o que foi observado no I09 (todas as 18 avaliações com nota 3), o quadro é de uso pro forma do instrumento: não há registros de inadequado, mas também não há diferenciação positiva. A ausência de nota 2 **não indica ausência de problemas** — pode indicar ausência de avaliação crítica.

### Nuances importantes para a interpretação

**Percentual baixo não garante ausência de problema:** uma unidade com 1 avaliação "Inadequado" em 3 totais tem `perc_inadequado = 33.33%` — alto alerta com base estatística muito fraca. Sempre observe `total_avaliacoes_pt` junto com o percentual.

**Percentual zero com nota única:** se `perc_inadequado = 0` e `total_avaliacoes_pt > 0`, verifique no I09 a distribuição das notas. Se todas as avaliações são nota 3 (como no caso AUDIT), o zero não é resultado de bom desempenho — é resultado de avaliação padronizada.

**Relação esperada entre I10 e I09:** em geral, unidades com I10 alto tendem a ter I09 baixo. Porém, uma unidade com distribuição bimodal (muitas notas 2 e muitas notas 5) pode ter I09 médio com I10 alto — o que indica polarização interna e não pode ser capturado apenas pela média.

**Leitura integrada com I12 (coerência):** se uma unidade tem `perc_inadequado` alto nos PTs mas a avaliação coletiva do PE é boa (nota alta), o I12 identificará essa divergência. Isso pode indicar que os avaliadores dos PTs estão aplicando critérios mais rigorosos do que a autoavaliação coletiva da unidade — ou que os problemas individuais não se refletem nos resultados coletivos da unidade.

### O que fazer quando o resultado estiver vazio

As mesmas causas do I09 se aplicam. Se `qtd_inadequado = 0` para todas as unidades (e não apenas para uma específica), verifique:

1. Se a escala tem nota com valor numérico `2` — use a Consulta 2 do mapeamento do Eixo 4 para confirmar.
2. Se as avaliações do período foram registradas com notas — use a Consulta 6 do mapeamento para ver a distribuição real.
