# I04 — Indice de Atingimento de Metas

## 1. Finalidade do indicador

O I04 mede o **score medio de execucao por unidade**, calculado como a media das taxas de atingimento de todas as entregas do ciclo.

Pergunta central:

**"Em media, qual percentual das metas planejadas a unidade executou no ciclo?"**

Diferenca principal para o I02:
- **I02:** criterio binario (concluiu ou nao concluiu)
- **I04:** criterio continuo (grau de execucao)

Por isso, **I04 costuma ser maior que I02**, e pode ultrapassar 100% quando as metas foram subestimadas no planejamento.

```python
"""IND_04.1_run.py
Serie temporal do I04 — Indice de Atingimento de Metas por Unidade.

O I04 calcula o score medio de execucao por unidade: media aritmetica das
proporcoes de atingimento (realizado / planejado) de todas as entregas do ciclo,
expressas em percentual. Cada entrega tem peso igual, independente do volume
absoluto da meta. O score pode ultrapassar 100% quando ha superexecucao.

Abordagem mista de ciclos (validada em 18/05/2026 para o I02, aplicada aqui
pois o I04 usa o mesmo escopo de PE):
  2025 → TRIMESTRAL  (T1: Jan-Mar | T2: Abr-Jun | T3: Jul-Set | T4: Out-Dez)
  2026 → QUADRIMESTRAL (Q1: Jan-Abr | Q2: Mai-Ago | Q3: Set-Dez)

Escopo identico ao I02: sobreposicao de datas do PE (nao data_fim individual
da entrega), garantindo que total_no_ciclo seja comparavel entre I02 e I04.

AVISO METODOLOGICO: scores acima de 100% indicam superexecucao e podem
refletir subestimacao das metas no planejamento, nao necessariamente
desempenho excepcional. A CGOV observou (A3 - 17/05/2026) que o indicador
pode ter variacoes por falta de registro de execucao em planos historicos
(periodo em que o sistema permitia conclusao sem o devido registro).

Validacao A3 (17/05/2026): formula confirmada para COGEP (dump 208,33% vs
PETRVS 208,25% — diferenca de 0,08 pp por precisao decimal).
"""
import csv
import os
import re
import sys
from datetime import date, datetime

import jpype
import jpype.imports

sys.stdout.reconfigure(encoding="utf-8", errors="replace")

# ── Conexao ─────────────────────────────────────────────────────────────────---
# Credenciais e caminhos lidos de .env para nao expor dados sensíveis
JAR = os.getenv("DENODO_JAR_PATH")
JVM_DLL = os.getenv("DENODO_JVM_DLL")
JDBC_URL = os.getenv("DENODO_JDBC_URL")
USER = os.getenv("DENODO_USER")
PASS = os.getenv("DENODO_PASS")

OUT_DIR = os.path.dirname(os.path.abspath(__file__))

SQL_I04 = """
WITH parametros AS (
    SELECT
        CAST('{ini}' AS DATE) AS data_inicio,
        CAST('{fim}' AS DATE) AS data_fim,
        0                     AS incluir_excluidos
),
universo_bruto AS (
    SELECT
        u.sigla     AS unidade_sigla,
        MIN(u.nome) AS unidade_nome,
        COUNT(*)    AS total_cadastradas
    FROM petrvs_icmbio_planos_entregas pe
    JOIN petrvs_icmbio_planos_entregas_entregas pee
        ON pee.plano_entrega_id = pe.id
    JOIN petrvs_icmbio_unidades u
        ON u.id = pe.unidade_id
    CROSS JOIN parametros p
    WHERE CAST(pe.data_inicio AS DATE) <= p.data_fim
      AND CAST(pe.data_fim   AS DATE) >= p.data_inicio
      AND (p.incluir_excluidos = 1 OR pe.deleted_at  IS NULL)
      AND (p.incluir_excluidos = 1 OR pee.deleted_at IS NULL)
    GROUP BY u.sigla
),
entregas_ciclo AS (
    SELECT
        u.sigla                                                   AS unidade_sigla,
        u.nome                                                    AS unidade_nome,
        pee.id                                                    AS id_entrega,
        ABS(COALESCE(pee.progresso_realizado, 0))
            / NULLIF(ABS(pee.progresso_esperado), 0)             AS proporcao_atingimento,
        CASE
            WHEN pe.status IN ('AVALIADO', 'CONCLUIDO') THEN 1 ELSE 0
        END                                                       AS plano_avaliado
    FROM petrvs_icmbio_planos_entregas pe
    JOIN petrvs_icmbio_planos_entregas_entregas pee
        ON pee.plano_entrega_id = pe.id
    JOIN petrvs_icmbio_unidades u
        ON u.id = pe.unidade_id
    CROSS JOIN parametros p
    WHERE CAST(pe.data_inicio AS DATE) <= p.data_fim
      AND CAST(pe.data_fim   AS DATE) >= p.data_inicio
      AND (p.incluir_excluidos = 1 OR pe.deleted_at  IS NULL)
      AND (p.incluir_excluidos = 1 OR pee.deleted_at IS NULL)
      AND pee.progresso_esperado IS NOT NULL
      AND pee.progresso_esperado > 0
),
resumo AS (
    SELECT
        unidade_sigla,
        MIN(unidade_nome)                                         AS unidade_nome,
        COUNT(id_entrega)                                         AS total_no_ciclo,
        ROUND(AVG(proporcao_atingimento) * 100.0, 2)             AS score_atingimento_perc,
        SUM(plano_avaliado)                                       AS total_em_plano_avaliado
    FROM entregas_ciclo
    GROUP BY unidade_sigla
)
SELECT
    r.unidade_sigla,
    r.unidade_nome,
    b.total_cadastradas,
    r.total_no_ciclo,
    r.score_atingimento_perc,
    r.total_em_plano_avaliado,
    CASE
        WHEN r.score_atingimento_perc >= 90  THEN 'A — Alto desempenho'
        WHEN r.score_atingimento_perc >= 70  THEN 'B — Bom desempenho'
        WHEN r.score_atingimento_perc >= 50  THEN 'C — Desempenho intermediario'
        ELSE                                      'D — Baixo desempenho'
    END AS grupo_performance,
    CASE
        WHEN r.total_no_ciclo > r.total_em_plano_avaliado
        THEN 'atencao: ha entregas em planos nao avaliados'
        ELSE 'ciclo avaliado'
    END AS alerta_avaliacao
FROM resumo r
LEFT JOIN universo_bruto b ON b.unidade_sigla = r.unidade_sigla
ORDER BY r.score_atingimento_perc DESC, r.unidade_sigla
"""


def build_periods():
    """2025 = trimestral | 2026+ = quadrimestral."""
    today = date.today()
    raw = [
        ("T1-2025", "trimestral",    date(2025, 1,  1), date(2025, 3, 31)),
        ("T2-2025", "trimestral",    date(2025, 4,  1), date(2025, 6, 30)),
        ("T3-2025", "trimestral",    date(2025, 7,  1), date(2025, 9, 30)),
        ("T4-2025", "trimestral",    date(2025, 10, 1), date(2025, 12, 31)),
        ("Q1-2026", "quadrimestral", date(2026, 1,  1), date(2026, 4, 30)),
        ("Q2-2026", "quadrimestral", date(2026, 5,  1), date(2026, 8, 31)),
        ("Q3-2026", "quadrimestral", date(2026, 9,  1), date(2026, 12, 31)),
    ]
    periods = []
    for lbl, tipo, p_ini, p_fim in raw:
        if p_ini > today:
            continue
        status = "em_andamento" if p_fim >= today else "encerrado"
        periods.append((lbl, tipo, p_ini, p_fim, status))
    return periods


def start_jvm():
    if not jpype.isJVMStarted():
        jpype.startJVM(JVM_DLL, classpath=[JAR])
        print("JVM iniciada.")


def get_conn():
    DM = jpype.JClass("java.sql.DriverManager")
    P  = jpype.JClass("java.util.Properties")()
    P.setProperty("user", USER)
    P.setProperty("password", PASS)
    return DM.getConnection(JDBC_URL, P)


def clean(val):
    if val is None:
        return ""
    return re.sub(r"[\r\n]+", " / ", str(val)).strip()


def run_period(conn, ini, fim):
    sql  = SQL_I04.format(ini=ini, fim=fim)
    stmt = conn.createStatement()
    rs   = stmt.executeQuery(sql)
    meta = rs.getMetaData()
    n    = meta.getColumnCount()
    cols = [str(meta.getColumnLabel(i + 1)) for i in range(n)]
    rows = []
    while rs.next():
        rows.append([clean(rs.getObject(i + 1)) for i in range(n)])
    rs.close()
    stmt.close()
    return cols, rows


def save_csv(path, cols, rows):
    with open(path, "w", newline="", encoding="utf-8-sig") as f:
        csv.writer(f, delimiter="|").writerows([cols] + rows)
    print(f"  Salvo: {os.path.basename(path)}")


def main():
    start_jvm()
    conn = get_conn()
    print("Conexao Denodo OK.\n")

    periods = build_periods()
    meta_cols = [
        "ciclo_tipo", "periodo", "periodo_inicio", "periodo_fim",
        "periodo_status", "duracao_dias",
    ]

    all_cols = None
    all_rows = []

    for lbl, tipo, p_ini, p_fim, status in periods:
        cols, rows = run_period(conn, p_ini, p_fim)
        if all_cols is None:
            all_cols = meta_cols + cols
        dur = (p_fim - p_ini).days + 1
        for row in rows:
            all_rows.append([tipo, lbl, str(p_ini), str(p_fim), status, dur] + row)

    conn.close()

    stamp = datetime.now().strftime("%Y%m%d_%H%M")
    fname = os.path.join(OUT_DIR, f"IND_04.2_score_atingimento_metas_{stamp}.csv")
    save_csv(fname, all_cols, all_rows)


if __name__ == "__main__":
    main()
```

## 3. Passos da consulta (explicacao detalhada)

1. **Definir os periodos e rotulos.**
    O script cria periodos com rotulos (T1-2025, Q1-2026 etc.) e marca se estao `encerrado` ou `em_andamento`. Use esses campos para decidir se um valor e definitivo.

2. **Construir o universo de entregas do ciclo (`entregas_ciclo`).**
    Seleciona entregas cujo PE se sobrepoe ao periodo e que possuem `progresso_esperado > 0`. Entregas sem meta valida sao excluidas para evitar divisao por zero.

3. **Calcular proporcao por entrega.**
    Para cada entrega, calcula-se `proporcao_atingimento = abs(progresso_realizado) / nullif(abs(progresso_esperado), 0)`. O valor absoluto evita problemas com sinais negativos e `NULLIF` previne divisao por zero.

4. **Agregacao por unidade (score medio).**
    O `score_atingimento_perc` e a media das proporcoes de todas as entregas do ciclo multiplicada por 100. Essa media reduz a variabilidade entre entregas e fornece um indicador agregador util para comparacao entre unidades.

5. **Classificacao e alertas.**
    A consulta adiciona `grupo_performance` (A/B/C/D) para facilitar dashboards e `alerta_avaliacao` para indicar se parte do ciclo ainda nao foi avaliada formalmente.

6. **Exportacao e metadados.**
    O CSV inclui colunas de metadado (periodo, duracao) para facilitar comparacoes historicas e o uso em relatórios gerenciais.

Use as tres consultas abaixo antes de gerar o I04.

### QT-01 — Distribuicao de planos de entrega por ciclo

```sql
SELECT
    EXTRACT(YEAR  FROM CAST(pe.data_inicio AS DATE)) AS ano,
    EXTRACT(MONTH FROM CAST(pe.data_inicio AS DATE)) AS mes_inicio,
    EXTRACT(MONTH FROM CAST(pe.data_fim   AS DATE))  AS mes_fim,
    COUNT(*)                                          AS total_planos,
    ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER (), 1) AS pct_total
FROM planos_entregas pe
WHERE pe.deleted_at IS NULL
  AND CAST(pe.data_inicio AS DATE) >= CAST('2025-01-01' AS DATE)
GROUP BY
    EXTRACT(YEAR  FROM CAST(pe.data_inicio AS DATE)),
    EXTRACT(MONTH FROM CAST(pe.data_inicio AS DATE)),
    EXTRACT(MONTH FROM CAST(pe.data_fim   AS DATE))
ORDER BY ano, mes_inicio
```

### QT-02 — Distribuicao de planos de trabalho por ciclo

```sql
SELECT
    EXTRACT(YEAR  FROM CAST(pt.data_inicio AS DATE)) AS ano,
    EXTRACT(MONTH FROM CAST(pt.data_inicio AS DATE)) AS mes_inicio,
    EXTRACT(MONTH FROM CAST(pt.data_fim   AS DATE))  AS mes_fim,
    COUNT(*)                                          AS total_planos,
    ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER (), 1) AS pct_total
FROM planos_trabalhos pt
WHERE pt.deleted_at IS NULL
  AND CAST(pt.data_inicio AS DATE) >= CAST('2025-01-01' AS DATE)
GROUP BY
    EXTRACT(YEAR  FROM CAST(pt.data_inicio AS DATE)),
    EXTRACT(MONTH FROM CAST(pt.data_inicio AS DATE)),
    EXTRACT(MONTH FROM CAST(pt.data_fim   AS DATE))
ORDER BY ano, mes_inicio
```

### QT-03 — Distribuicao de avaliacoes por mes de referencia

```sql
SELECT
    EXTRACT(YEAR  FROM CAST(a.data_avaliacao AS DATE)) AS ano,
    EXTRACT(MONTH FROM CAST(a.data_avaliacao AS DATE)) AS mes,
    COUNT(*)                                            AS total_avaliacoes
FROM avaliacoes a
WHERE a.deleted_at IS NULL
  AND CAST(a.data_avaliacao AS DATE) >= CAST('2025-01-01' AS DATE)
GROUP BY
    EXTRACT(YEAR  FROM CAST(a.data_avaliacao AS DATE)),
    EXTRACT(MONTH FROM CAST(a.data_avaliacao AS DATE))
ORDER BY ano, mes
```

## 7. Como interpretar o resultado (exemplos reais)

### Como interpretar os exemplos (guia para gestores)

Ao analisar um `score_atingimento_perc`, um gestor deve verificar em ordem:

1. **`periodo_status`** — se `em_andamento`, trate o valor como provisório.
2. **`total_no_ciclo`** — scores com bases muito pequenas (ex: < 5 entregas) sao menos confiaveis.
3. **`total_cadastradas` vs `total_no_ciclo`** — diferencas grandes indicam problemas de cadastro de metas.
4. **`alerta_avaliacao`** — se ha entregas nao avaliadas, adie decisoes formais.
5. **Analise qualitativa** — abrir amostra de `id_entrega` e `nome_entrega` para inspecao qualitativa quando o score for muito alto/baixo.

Exemplos abaixo usam o CSV `IND_04.2_score_atingimento_metas_20260524_1301.csv`.

### Exemplo 1 — Score muito acima de 100%

No periodo **T1-2025**, a unidade **NGIBSBCONTAGEM** teve:

- `score_atingimento_perc = 2017.20`

Interpretacao gerencial: isso indica metas muito subestimadas no planejamento. Antes de celebrar, o gestor deve:

- Verificar `total_no_ciclo` (quantas entregas sustentam esse score).
- Abrir amostra de entregas para verificar se ha apenas 1 ou 2 entregas com valor extremamente alto que distorcem a media.
- Se a base for grande e o padrao constante, ajustar metas no ciclo seguinte e registrar a justificativa no relatorio de validacao.

## 8. Observacoes de validacao e limitacoes metodologicas

### 8.1 Validacao metodologica (Fase 7 — 17/05/2026)

A fórmula do I04 foi validada pela equipe CGOV comparando o dump MySQL (fev/2026) com dados observados no sistema PETRVS para a unidade COGEP:

| Fonte | Score atingimento | Diferenca |
| --- | --- | --- |
| Dump (IND_04 calculo anterior) | 208,33% | — |
| PETRVS (consulta manual CGOV) | 208,25% | 0,08 pp |

A diferenca de 0,08 pp é atribuida a variabilidade na precisao decimal (DECIMAL(5,2)) durante processos intermediarios. **A fórmula foi confirmada sem necessidade de ajuste.**

Documento de validacao: [Testes PETRVS/IND_04.5_relatorio_validacao_24.05.2026.md](../Testes PETRVS/IND_04.5_relatorio_validacao_24.05.2026.md)

### 8.2 Interpretacao de scores acima de 100%

Scores acima de 100% **sao legítimos e nao representam erro**. Indicam que:

1. Uma ou mais entregas foram executadas com sucesso bem acima da meta planejada (ex: 200%, 333%).
2. As metas foram **sistematicamente subestimadas** no planejamento.
3. Este é um sinal gerencial relevante para o planejador revisar metodologia de estimativa no ciclo seguinte.

**Nao interpretar como "desempenho excepcional"** — ao inves, agende reuniao com a unidade para discutir se:
- As metas estavam irrealistas?
- Houve mudanca de escopo ou recursos durante o ciclo?
- A execucao se beneficiou de fatores externos nao previstos?

### 8.3 Impacto de ciclos diferentes (2025 vs 2026+)

A regra temporal do I04 acompanha a do I02:
- **2025:** trimestral (T1, T2, T3, T4)
- **2026+:** quadrimestral (Q1, Q2, Q3)

Essa mudanca implica que os periodos nao sao comparaveis entre si:
- T1–T4 de 2025: 3 meses cada
- Q1–Q3 de 2026: ~4.3 meses cada

**Nunca compare T4-2025 com Q1-2026.** Para analise longitudinal, use totais anuais (2025 inteiro vs. 2026 inteiro).

### 8.4 Impacto de entregas sem meta valida

O I04 exclui entregas sem `progresso_esperado > 0` do denominador. Isso significa:

- Entrega cadastrada mas sem meta: nao entra no `total_no_ciclo`.
- Diferencas grandes entre `total_cadastradas` e `total_no_ciclo`: sinal de qualidade deficiente.
- Score pode ser inflado se muitas entregas sao omitidas por falta de meta.

A coluna `total_cadastradas` permite ao gestor detectar esse problema. Se divergencia > 10%, o script A1 emite aviso automático.

### 8.5 Impacto de planos nao avaliados

A coluna `alerta_avaliacao` identifica quando parte do ciclo ainda nao recebeu avaliacao formal:

```
alerta_avaliacao = 'atencao: ha entregas em planos nao avaliados'
```

Nesses casos, o score é provisório. Aguarde a conclusao da avaliacao formal antes de usar o resultado em relatorios gerenciais definitivos.

### 8.6 Variabilidade historica (falta de registro em planos antigos)

A equipe CGOV observou durante a validacao que alguns planos historicos (2024–inicio 2025) foram concluidos sem registro formal de execucao das entregas. Isso pode causar:

- `progresso_realizado = 0` ou NULL para entregas na verdade executadas.
- Scores mais baixos que a realidade para esses periodos.

Com Denodo em tempo real, esse impacto é menor para dados atuais (mai/2026+), mas nao nulo para dados historicos. Use a coluna `periodo_status` para filtrar apenas periodos `encerrado` ao fazer analises históricas comparativas.
