# I05 — Distribuição das Entregas entre os Servidores

## 1. Finalidade do indicador

O I05 calcula **quantas entregas do Plano de Entregas cada servidor carrega no seu Plano de Trabalho**, e compara essa quantidade com a média dos demais servidores da mesma unidade.

A pergunta central que o I05 responde é: **"A carga de entregas está distribuída de forma equitativa entre os servidores da unidade, ou está concentrada em poucos?"**

O resultado traz uma linha por servidor com:

- **`qtd_entregas_por_servidor`** — quantas entregas distintas esse servidor tem vinculadas no período
- **`media_entregas_por_servidor_unidade`** — a média de entregas por servidor dentro da unidade (calculada via window function)
- **`posicao_relativa_media`** — classificação: "Acima da media", "Na media" ou "Abaixo da media"

O I05 **não mede se as entregas foram cumpridas** — isso é o Eixo 2. O que ele mede é a distribuição do esforço planejado: quantos compromissos cada servidor assumiu formalmente no período.

**Perspectiva complementar:** o I05 olha a distribuição a partir do servidor ("quantas entregas tem cada pessoa?"). O I06 olha o mesmo vínculo a partir da entrega ("quantos servidores tem cada entrega?"). A leitura dos dois juntos fecha o diagnóstico de carga.

## 2. Consulta completa (MySQL)

```sql
with parametros as (
    select
        date('2025-01-01') as data_inicio,
        date('2025-12-31') as data_fim,
        0 as incluir_excluidos
),
vinculos_entregas as (
    select distinct
        coalesce(un.sigla, 'N.I.') as unidade_sigla,
        coalesce(un.nome, 'N.I.')  as unidade_nome,
        pt.usuario_id              as id_servidor,
        coalesce(us.nome, 'N.I.')  as nome_servidor,
        pte.plano_entrega_entrega_id as id_entrega
    from planos_trabalhos pt
    join planos_trabalhos_entregas pte
        on pte.plano_trabalho_id = pt.id
    left join unidades un
        on un.id = pt.unidade_id
    left join usuarios us
        on us.id = pt.usuario_id
    cross join parametros p
    where date(pt.data_inicio) <= p.data_fim
      and date(pt.data_fim)   >= p.data_inicio
      and (p.incluir_excluidos = 1 or pt.deleted_at is null)
      and pt.usuario_id is not null
      and pte.plano_entrega_entrega_id is not null
),
entregas_por_servidor as (
    select
        unidade_sigla,
        min(unidade_nome)              as unidade_nome,
        id_servidor,
        min(nome_servidor)             as nome_servidor,
        count(distinct id_entrega)     as qtd_entregas_por_servidor
    from vinculos_entregas
    group by unidade_sigla, id_servidor
),
com_media as (
    select
        unidade_sigla,
        unidade_nome,
        id_servidor,
        nome_servidor,
        qtd_entregas_por_servidor,
        round(
            avg(qtd_entregas_por_servidor) over (partition by unidade_sigla),
            2
        ) as media_entregas_por_servidor_unidade
    from entregas_por_servidor
)
select
    unidade_sigla,
    unidade_nome,
    id_servidor,
    nome_servidor,
    qtd_entregas_por_servidor,
    media_entregas_por_servidor_unidade,
    case
        when qtd_entregas_por_servidor > media_entregas_por_servidor_unidade then 'Acima da media'
        when qtd_entregas_por_servidor < media_entregas_por_servidor_unidade then 'Abaixo da media'
        else 'Na media'
    end as posicao_relativa_media
from com_media
order by unidade_sigla, qtd_entregas_por_servidor desc, nome_servidor;
```

> **Requisito técnico:** esta consulta usa `AVG() OVER (PARTITION BY)`, uma window function disponível apenas no **MySQL 8.0 ou superior**. Confirme a versão com `SELECT VERSION();` antes de executar.

## 3. Passos da consulta

A consulta tem quatro CTEs encadeadas antes do `SELECT` final.

### Passo 1 — Bloco `parametros`: painel de controle

```sql
parametros as (
    select
        date('2025-01-01') as data_inicio,
        date('2025-12-31') as data_fim,
        0 as incluir_excluidos
)
```

Ajuste `data_inicio` e `data_fim` para o período que deseja analisar. O filtro é aplicado sobre a **vigência do Plano de Trabalho** — não sobre a data de conclusão de cada entrega. Um plano ativo entre outubro e dezembro será incluído numa análise de julho a dezembro, pois há sobreposição de vigências.

O `incluir_excluidos` controla o soft-delete:

- `0` (padrão): apenas planos ativos (`deleted_at is null`)
- `1`: inclui planos excluídos logicamente — útil para auditoria histórica

---

### Passo 2 — Bloco `vinculos_entregas`: pares únicos servidor-entrega

```sql
vinculos_entregas as (
    select distinct
        coalesce(un.sigla, 'N.I.') as unidade_sigla,
        coalesce(un.nome, 'N.I.')  as unidade_nome,
        pt.usuario_id              as id_servidor,
        coalesce(us.nome, 'N.I.')  as nome_servidor,
        pte.plano_entrega_entrega_id as id_entrega
    from planos_trabalhos pt
    join planos_trabalhos_entregas pte
        on pte.plano_trabalho_id = pt.id
    left join unidades un on un.id = pt.unidade_id
    left join usuarios us on us.id = pt.usuario_id
    cross join parametros p
    where date(pt.data_inicio) <= p.data_fim
      and date(pt.data_fim)   >= p.data_inicio
      and (p.incluir_excluidos = 1 or pt.deleted_at is null)
      and pt.usuario_id is not null
      and pte.plano_entrega_entrega_id is not null
)
```

Este bloco percorre o caminho de dados: `planos_trabalhos` → `planos_trabalhos_entregas`, coletando os pares (servidor, entrega) ativos no período.

**Por que `SELECT DISTINCT`:** um servidor pode ter múltiplas linhas na tabela `planos_trabalhos_entregas` para a mesma entrega caso o plano tenha sido editado ou houver duplicatas de cadastro. O `DISTINCT` garante que cada par (servidor, entrega) seja contado uma única vez, independentemente de quantas linhas existam na tabela de vínculo.

**Por que `JOIN` (não `LEFT JOIN`) em `planos_trabalhos_entregas`:** só nos interessam planos que têm entregas vinculadas. Planos de trabalho sem nenhuma entrega associada não produzem linha aqui — e corretamente, pois não há carga a medir.

**Filtros aplicados:**

- `date(pt.data_inicio) <= p.data_fim AND date(pt.data_fim) >= p.data_inicio`: seleciona planos com vigência que se sobrepõe ao período, mesmo que parcialmente.
- `pt.usuario_id is not null`: exclui planos sem servidor identificado — eles não poderiam ser atribuídos a ninguém.
- `pte.plano_entrega_entrega_id is not null`: exclui vínculos com entrega não identificada.

**`LEFT JOIN` em `unidades` e `usuarios`:** permite que servidores e unidades sem cadastro completo apareçam como `'N.I.'` em vez de serem silenciosamente descartados do resultado.

---

### Passo 3 — Bloco `entregas_por_servidor`: contagem por servidor

```sql
entregas_por_servidor as (
    select
        unidade_sigla,
        min(unidade_nome)          as unidade_nome,
        id_servidor,
        min(nome_servidor)         as nome_servidor,
        count(distinct id_entrega) as qtd_entregas_por_servidor
    from vinculos_entregas
    group by unidade_sigla, id_servidor
)
```

Agrupa os pares (servidor, entrega) por servidor dentro de cada unidade e conta quantas entregas distintas cada servidor carrega.

**`count(distinct id_entrega)`:** o `DISTINCT` aqui é uma proteção adicional. Mesmo que o bloco anterior já use `SELECT DISTINCT`, diferenças de `unidade_sigla` (ex: `'N.I.'` vs sigla real) em linhas diferentes para o mesmo par poderiam gerar duplicatas no agrupamento. O `DISTINCT` no `COUNT` elimina esse risco residual.

**`min(unidade_nome)` e `min(nome_servidor)`:** o `GROUP BY` usa `unidade_sigla` e `id_servidor` (chaves estáveis). Os campos de texto (nome da unidade, nome do servidor) são trazidos com `MIN()` apenas para satisfazer a sintaxe do `GROUP BY` do MySQL — como cada `id_servidor` pertence a uma unidade, o valor de `min()` será o único nome disponível.

---

### Passo 4 — Bloco `com_media`: média da unidade via window function

```sql
com_media as (
    select
        unidade_sigla,
        unidade_nome,
        id_servidor,
        nome_servidor,
        qtd_entregas_por_servidor,
        round(
            avg(qtd_entregas_por_servidor) over (partition by unidade_sigla),
            2
        ) as media_entregas_por_servidor_unidade
    from entregas_por_servidor
)
```

Este é o bloco que exige MySQL 8.0+. A window function calcula a média de `qtd_entregas_por_servidor` para todos os servidores da mesma `unidade_sigla`, e **repete esse valor em cada linha** — sem colapsar as linhas individuais.

A diferença em relação a um `GROUP BY` convencional: o `GROUP BY` produziria uma linha por unidade com a média; a window function mantém uma linha por servidor e acrescenta a média ao lado, permitindo a comparação individual diretamente no `SELECT` final.

**Exemplo numérico:** se a CGOV tem três servidores com 2, 4 e 6 entregas, a window function retorna `media = 4.00` em todas as três linhas — e o `SELECT` final classifica: 2 < 4 = "Abaixo da media", 4 = 4 = "Na media", 6 > 4 = "Acima da media".

---

### Passo 5 — SELECT final: classificação e ordenação

```sql
select
    unidade_sigla,
    unidade_nome,
    id_servidor,
    nome_servidor,
    qtd_entregas_por_servidor,
    media_entregas_por_servidor_unidade,
    case
        when qtd_entregas_por_servidor > media_entregas_por_servidor_unidade then 'Acima da media'
        when qtd_entregas_por_servidor < media_entregas_por_servidor_unidade then 'Abaixo da media'
        else 'Na media'
    end as posicao_relativa_media
from com_media
order by unidade_sigla, qtd_entregas_por_servidor desc, nome_servidor;
```

Aplica a classificação em três faixas e ordena por unidade, depois por quantidade decrescente de entregas — os servidores mais carregados aparecem primeiro dentro de cada unidade. O desempate final por `nome_servidor` garante ordenação determinística entre servidores com a mesma quantidade.

**A classificação é relativa, não absoluta:** "Acima da media" em uma unidade com média 2 significa uma coisa muito diferente de "Acima da media" em uma unidade com média 10. A coluna `media_entregas_por_servidor_unidade` sempre aparece ao lado de `posicao_relativa_media` exatamente para que o analista veja o valor de referência usado na comparação.

## 4. Como interpretar o resultado

### Estrutura do resultado

| Coluna | O que informa |
| --- | --- |
| `unidade_sigla` | Sigla da unidade — agrupa o resultado |
| `unidade_nome` | Nome completo da unidade |
| `id_servidor` | Identificador do servidor na base (campo `usuario_id`) |
| `nome_servidor` | Nome do servidor conforme cadastrado em `usuarios` |
| `qtd_entregas_por_servidor` | Quantas entregas distintas esse servidor tem vinculadas no período |
| `media_entregas_por_servidor_unidade` | Média de entregas por servidor calculada dentro da unidade |
| `posicao_relativa_media` | "Acima da media", "Na media" ou "Abaixo da media" em relação à média da unidade |

### Exemplo com dados do ICMBio

Análise do período 2025-01-01 a 2025-12-31. Cada linha representa um servidor:

| unidade_sigla | nome_servidor | qtd_entregas_por_servidor | media_entregas_por_servidor_unidade | posicao_relativa_media |
| --- | --- | --- | --- | --- |
| CGOV | Ana Lima | 9 | 4.00 | Acima da media |
| CGOV | Bruno Costa | 4 | 4.00 | Na media |
| CGOV | Carla Souza | 4 | 4.00 | Na media |
| CGOV | Diego Alves | 3 | 4.00 | Abaixo da media |
| CGOV | Eliane Faria | 0 | 4.00 | Abaixo da media |
| AUDIT | Fábio Nunes | 6 | 3.00 | Acima da media |
| AUDIT | Gisele Moura | 3 | 3.00 | Na media |
| AUDIT | Hugo Teixeira | 0 | 3.00 | Abaixo da media |
| COGEO | Irene Castro | 3 | 3.00 | Na media |
| COGEO | Jonas Melo | 3 | 3.00 | Na media |
| COGEO | Karen Pinto | 3 | 3.00 | Na media |

**Leituras caso a caso:**

**CGOV — Ana Lima (9 entregas, média 4.00):** Ana carrega mais que o dobro da média da unidade. Isso pode indicar: (a) perfil de servidor de referência para muitas entregas da unidade; (b) acúmulo informal de funções; (c) problema de cadastro onde as entregas foram vinculadas ao plano dela por padrão. O I06 complementa a análise — se as 9 entregas da Ana também têm outros servidores vinculados, o risco é distribuído; se ela é a única responsável em várias delas, há concentração real.

**CGOV — Eliane Faria (0 entregas, média 4.00):** servidor ativo na unidade no período, mas sem nenhuma entrega vinculada ao plano de trabalho. Pode indicar: plano de trabalho mal preenchido (o servidor existe mas não vinculou entregas), servidor em fase de integração, ou afastamento prolongado com plano cadastrado mas não operacional. Vale cruzar com o I01 (regime de trabalho) para verificar se o servidor estava efetivamente em atividade no período.

**AUDIT — Hugo Teixeira (0 entregas, média 3.00):** mesma situação de Eliane. Na AUDIT, com apenas 3 servidores, um servidor sem entregas representa 33% da equipe sem carga registrada — impacto maior do que numa unidade grande.

**COGEO — distribuição equilibrada (todos com 3, média 3.00):** caso de distribuição homogênea. Todos os servidores da unidade carregam o mesmo número de entregas, e a média reflete exatamente essa igualdade. O I06 pode revelar se essas 3 entregas de cada servidor são as mesmas (todos vinculados às mesmas entregas = responsabilidade coletiva) ou entregas distintas (cada servidor tem um portfólio individual).

### Nuances importantes

**A média é calculada sobre servidores com pelo menos um plano ativo no período, não sobre o total de servidores da unidade.** Servidores sem plano de trabalho ativo não aparecem no resultado — o que significa que a "média da unidade" reflete apenas quem tem plano cadastrado, não a lotação total.

**Servidores com 0 entregas aparecem no resultado** porque têm plano de trabalho ativo no período, mas sem vínculos com entregas. Eles puxam a média para baixo e são classificados como "Abaixo da media". Sua presença no resultado é informativa — indica lacuna de cadastro ou situação funcional especial.

**A média inclui os servidores com 0 entregas no denominador.** Isso é intencional: a média deve refletir a realidade da distribuição, inclusive os servidores que não foram vinculados a nenhuma entrega.

**Uma unidade com um único servidor terá `posicao_relativa_media = 'Na media'`** sempre, pois a média é calculada sobre um único valor. Isso é matematicamente correto e não indica equilíbrio real — apenas que não há outros servidores para comparar.

### Leitura integrada com outros indicadores

| Situação observada no I05 | Próximo indicador | Pergunta a responder |
| --- | --- | --- |
| Servidor muito acima da média | I06 | As entregas dele têm outros responsáveis, ou ele é o único em todas? |
| Servidor com 0 entregas | I01 | O servidor estava em teletrabalho integral? Há plano cadastrado? |
| Unidade com média muito alta (ex: > 10) | I07 | Quantas horas totais isso representa? A carga é proporcional ao tempo disponível? |
| Distribuição muito desigual (alguns com 0, outros com 10+) | I02 | As entregas concentradas nos sobrecarregados têm menor taxa de cumprimento? |
