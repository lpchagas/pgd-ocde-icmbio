# I09 — Média da Avaliação do Plano de Trabalho por Unidade

## 1. Finalidade do indicador

O I09 calcula a **nota média das avaliações formais dos Planos de Trabalho (PT)** de cada unidade organizacional no período analisado.

Enquanto os indicadores do Eixo 2 (I02-I04) medem se as entregas foram concluídas com base em dados objetivos de progresso, o I09 mede **como os próprios atores avaliaram a qualidade do trabalho realizado**. É a voz do sistema de avaliação formal do PGD traduzida em número.

A pergunta central que o I09 responde é: **"Em média, qual foi a nota que os Planos de Trabalho dos servidores receberam nas avaliações formais do PETRVS?"**

O resultado é uma linha por unidade com a média das notas, a distribuição de avaliações em cada nota da escala (1 a 5) e uma classificação de faixa de desempenho. Esse conjunto permite:

- Comparar o desempenho médio entre unidades;
- Identificar unidades com padrão de avaliação preocupante (concentração em notas 1-2) ou de destaque (concentração em notas 4-5);
- Verificar se a escala está sendo usada de forma diferenciada (distribuição variada) ou de forma pro forma (todos concentrados na nota 3);
- Alimentar análises de coerência com o I12 (PT vs PE).

> **Pré-requisito:** antes de executar esta consulta, execute as consultas de mapeamento do [Eixo 4](06.4-eixo4.md) para confirmar os nomes dos campos das tabelas de avaliação no seu banco.

## 2. Consulta completa (MySQL)

```sql
with parametros as (
    select
        date('2025-01-01') as data_inicio,
        date('2025-12-31') as data_fim,
        0                  as incluir_excluidos
),
avaliacoes_pt as (
    select
        av.id          as id_avaliacao,
        pt.unidade_id,
        case TRIM(BOTH '"' FROM JSON_UNQUOTE(tan.nota))
            when 'Não executado'   then 1
            when 'Inadequado'      then 2
            when 'Adequado'        then 3
            when 'Alto desempenho' then 4
            when 'Excepcional'     then 5
        end                        as valor_nota
    from avaliacoes av
    join planos_trabalhos_consolidacoes ptc
        on ptc.id = av.plano_trabalho_consolidacao_id
    join planos_trabalhos pt
        on pt.id = ptc.plano_trabalho_id
    join tipos_avaliacoes_notas tan
        on tan.id = av.tipo_avaliacao_nota_id
    cross join parametros p
    where av.plano_trabalho_consolidacao_id is not null
      and (p.incluir_excluidos = 1 or av.deleted_at is null)
      and date(pt.data_inicio) <= p.data_fim
      and date(pt.data_fim)   >= p.data_inicio
      and (p.incluir_excluidos = 1 or pt.deleted_at is null)
),
media_por_unidade as (
    select
        coalesce(un.sigla, 'N.I.')                               as unidade_sigla,
        coalesce(un.nome, 'N.I.')                                as unidade_nome,
        count(avpt.id_avaliacao)                                 as total_avaliacoes_pt,
        round(avg(avpt.valor_nota), 2)                           as media_nota_pt,
        min(avpt.valor_nota)                                     as nota_minima,
        max(avpt.valor_nota)                                     as nota_maxima,
        sum(case when avpt.valor_nota = 1 then 1 else 0 end)     as qtd_nota_1,
        sum(case when avpt.valor_nota = 2 then 1 else 0 end)     as qtd_nota_2,
        sum(case when avpt.valor_nota = 3 then 1 else 0 end)     as qtd_nota_3,
        sum(case when avpt.valor_nota = 4 then 1 else 0 end)     as qtd_nota_4,
        sum(case when avpt.valor_nota = 5 then 1 else 0 end)     as qtd_nota_5
    from avaliacoes_pt avpt
    left join unidades un
        on un.id = avpt.unidade_id
    group by coalesce(un.sigla, 'N.I.'), coalesce(un.nome, 'N.I.')
)
select
    unidade_sigla,
    unidade_nome,
    total_avaliacoes_pt,
    media_nota_pt,
    nota_minima,
    nota_maxima,
    qtd_nota_1,
    qtd_nota_2,
    qtd_nota_3,
    qtd_nota_4,
    qtd_nota_5,
    case
        when media_nota_pt >= 4.5 then 'Excepcional'
        when media_nota_pt >= 3.5 then 'Alto desempenho'
        when media_nota_pt >= 2.5 then 'Adequado'
        when media_nota_pt >= 1.5 then 'Inadequado'
        else 'Não executado'
    end as faixa_desempenho
from media_por_unidade
order by media_nota_pt desc, unidade_sigla;
```

## 3. Passos da consulta

A consulta é organizada em três blocos `WITH` encadeados antes do `SELECT` final. Cada bloco resolve um subproblema específico.

### Passo 1 — Bloco `parametros`: painel de controle da consulta

```sql
parametros as (
    select
        date('2025-01-01') as data_inicio,
        date('2025-12-31') as data_fim,
        0                  as incluir_excluidos
)
```

**O que faz:** define o período de análise e o comportamento para registros excluídos logicamente. É o único bloco que você precisa editar para ajustar a consulta.

- `data_inicio` e `data_fim`: filtram os planos de trabalho cuja vigência se sobrepõe ao período. Um plano que começou em outubro/2024 e terminou em março/2025 seria incluído em uma análise de 2025 — porque parte do plano ocorreu no período.
- `incluir_excluidos = 0`: considera apenas avaliações e planos ativos (`deleted_at is null`). Use `1` para auditorias históricas que precisam incluir registros excluídos logicamente.

**Analogia:** é como configurar os filtros de um painel de BI antes de abrir o relatório — você define o recorte temporal uma vez, e todo o resto da consulta se ajusta automaticamente.

---

### Passo 2 — Bloco `avaliacoes_pt`: coletando as avaliações de Plano de Trabalho

Este é o bloco mais importante da consulta. Ele percorre a cadeia de relacionamentos do PETRVS para identificar, de cada avaliação registrada, **qual unidade está sendo avaliada** e **qual foi a nota**.

```sql
avaliacoes_pt as (
    select
        av.id          as id_avaliacao,
        pt.unidade_id,
        case TRIM(BOTH '"' FROM JSON_UNQUOTE(tan.nota))
            when 'Não executado'   then 1
            when 'Inadequado'      then 2
            when 'Adequado'        then 3
            when 'Alto desempenho' then 4
            when 'Excepcional'     then 5
        end                        as valor_nota
    from avaliacoes av
    join planos_trabalhos_consolidacoes ptc
        on ptc.id = av.plano_trabalho_consolidacao_id
    join planos_trabalhos pt
        on pt.id = ptc.plano_trabalho_id
    join tipos_avaliacoes_notas tan
        on tan.id = av.tipo_avaliacao_nota_id
    cross join parametros p
    where av.plano_trabalho_consolidacao_id is not null
      and (p.incluir_excluidos = 1 or av.deleted_at is null)
      and date(pt.data_inicio) <= p.data_fim
      and date(pt.data_fim)   >= p.data_inicio
      and (p.incluir_excluidos = 1 or pt.deleted_at is null)
)
```

**Por que a cadeia é necessária:**

No PETRVS, uma avaliação não é vinculada diretamente ao plano de trabalho do servidor. O caminho é:

```
avaliacoes  →  planos_trabalhos_consolidacoes  →  planos_trabalhos  →  unidade_id
```

- `avaliacoes`: cada registro é uma avaliação concreta. O campo `plano_trabalho_consolidacao_id` indica que esta avaliação pertence ao fechamento de um Plano de Trabalho. Avaliações com este campo preenchido são as de PT; as demais são de PE ou de outro tipo.
- `planos_trabalhos_consolidacoes`: é o "encerramento formal" de um ciclo de plano de trabalho. Cada consolidação pertence a um plano de trabalho (`plano_trabalho_id`).
- `planos_trabalhos`: o plano de trabalho contém `unidade_id` — a unidade a que o servidor pertence. É aqui que a avaliação individual ganha uma identidade organizacional.
- `tipos_avaliacoes_notas`: contém a escala 1-5. O join busca a nota a partir do identificador registrado na avaliação (`av.tipo_avaliacao_nota_id`). O campo `tan.nota` é do tipo JSON e armazena o rótulo textual da nota (`'Adequado'`, `'Excepcional'`, etc.). O `JSON_UNQUOTE()` extrai o texto puro, e o bloco `case` converte esse rótulo em valor numérico (1 a 5) para viabilizar os cálculos de média e distribuição.

**O filtro `plano_trabalho_consolidacao_id is not null`:** este é o critério que separa as avaliações de PT das demais. Avaliações de Plano de Entregas (PE) têm `plano_entrega_id` preenchido e `plano_trabalho_consolidacao_id` nulo — e são excluídas aqui deliberadamente, pois serão usadas no I12.

**O filtro de período** (`date(pt.data_inicio) <= p.data_fim and date(pt.data_fim) >= p.data_inicio`): seleciona planos de trabalho cuja vigência tem qualquer sobreposição com o período de análise — não apenas planos que começaram e terminaram dentro do intervalo.

### Passo 3 — Bloco `media_por_unidade`: calculando a média e a distribuição por unidade

```sql
media_por_unidade as (
    select
        coalesce(un.sigla, 'N.I.')                               as unidade_sigla,
        coalesce(un.nome, 'N.I.')                                as unidade_nome,
        count(avpt.id_avaliacao)                                 as total_avaliacoes_pt,
        round(avg(avpt.valor_nota), 2)                           as media_nota_pt,
        min(avpt.valor_nota)                                     as nota_minima,
        max(avpt.valor_nota)                                     as nota_maxima,
        sum(case when avpt.valor_nota = 1 then 1 else 0 end)     as qtd_nota_1,
        sum(case when avpt.valor_nota = 2 then 1 else 0 end)     as qtd_nota_2,
        sum(case when avpt.valor_nota = 3 then 1 else 0 end)     as qtd_nota_3,
        sum(case when avpt.valor_nota = 4 then 1 else 0 end)     as qtd_nota_4,
        sum(case when avpt.valor_nota = 5 then 1 else 0 end)     as qtd_nota_5
    from avaliacoes_pt avpt
    left join unidades un
        on un.id = avpt.unidade_id
    group by coalesce(un.sigla, 'N.I.'), coalesce(un.nome, 'N.I.')
)
```

**O que faz:** agrupa todas as avaliações de PT do passo anterior por unidade e calcula, para cada unidade:

- `count(id_avaliacao)`: quantas avaliações de PT existem no período. Número importante para avaliar a representatividade da média — uma média de 4.0 com base em 3 avaliações tem peso estatístico muito menor que a mesma média com 80 avaliações.
- `avg(valor_nota)`: a média aritmética simples das notas. O `round(..., 2)` limita a 2 casas decimais.
- `min` e `max`: amplitudes da escala usada na unidade. Uma unidade com `nota_minima = 1` e `nota_maxima = 5` usa a escala completa; uma com `nota_minima = 3` e `nota_maxima = 3` teve todas as avaliações concentradas no mesmo valor.
- Os campos `qtd_nota_1` a `qtd_nota_5`: contagem de avaliações em cada patamar da escala. Cada `case when` funciona como um "se" no Excel — conta 1 se a nota for igual ao valor testado, 0 caso contrário. A soma de todos os `sum(case when...)` deve ser igual a `total_avaliacoes_pt`.

**O `left join unidades`:** garante que avaliações vinculadas a unidades sem cadastro na tabela `unidades` não sejam perdidas — elas aparecem com sigla e nome como `'N.I.'` (Não Informado).

**O `group by`:** agrupa os cálculos por unidade. O uso de `coalesce(un.sigla, 'N.I.')` diretamente no `GROUP BY` garante que o agrupamento funcione mesmo para avaliações sem unidade identificada.

---

### Passo 4 — SELECT final: classificando por faixa de desempenho

```sql
select
    unidade_sigla,
    unidade_nome,
    total_avaliacoes_pt,
    media_nota_pt,
    nota_minima,
    nota_maxima,
    qtd_nota_1,
    qtd_nota_2,
    qtd_nota_3,
    qtd_nota_4,
    qtd_nota_5,
    case
        when media_nota_pt >= 4.5 then 'Excepcional'
        when media_nota_pt >= 3.5 then 'Alto desempenho'
        when media_nota_pt >= 2.5 then 'Adequado'
        when media_nota_pt >= 1.5 then 'Inadequado'
        else 'Não executado'
    end as faixa_desempenho
from media_por_unidade
order by media_nota_pt desc, unidade_sigla;
```

**O bloco `case` de faixa:** traduz a média numérica em uma categoria qualitativa. Os limites foram definidos como ponto médio entre notas adjacentes da escala:

| Faixa | Condição | Significado |
| --- | --- | --- |
| Excepcional | `media >= 4.5` | Média próxima a 5 — padrão de excelência |
| Alto desempenho | `media >= 3.5` | Média entre 4 e 5 — acima do esperado |
| Adequado | `media >= 2.5` | Média entre 3 e 4 — dentro do esperado |
| Inadequado | `media >= 1.5` | Média entre 2 e 3 — abaixo do esperado |
| Não executado | `media < 1.5` | Média próxima a 1 — padrão crítico |

**Por que separar em CTE antes de classificar:** em MySQL, não é possível referenciar um alias calculado (`media_nota_pt`) dentro do mesmo `SELECT` onde ele foi definido. Separar o cálculo da média no bloco `media_por_unidade` e a classificação no `SELECT` final resolve esse limite de forma limpa.

**A ordenação `order by media_nota_pt desc, unidade_sigla`:** apresenta primeiro as unidades com maior nota média. Em caso de empate na média, ordena alfabeticamente pela sigla.

## 4. Como interpretar o resultado

### Estrutura do resultado

Cada linha representa uma unidade organizacional do ICMBio com Planos de Trabalho avaliados no período. As colunas principais são:

| Coluna | O que informa |
| --- | --- |
| `unidade_sigla` / `unidade_nome` | Identidade da unidade |
| `total_avaliacoes_pt` | Quantas avaliações de PT foram consideradas |
| `media_nota_pt` | Média das notas (1 a 5) — o indicador central |
| `nota_minima` / `nota_maxima` | Amplitude da escala usada na unidade |
| `qtd_nota_1` a `qtd_nota_5` | Distribuição das avaliações por nota |
| `faixa_desempenho` | Classificação qualitativa da média |

### Exemplo com dados do ICMBio

Resultado hipotético de uma análise de 2025 com 5 unidades:

| unidade_sigla | total_avaliacoes_pt | media_nota_pt | nota_min | nota_max | qtd_1 | qtd_2 | qtd_3 | qtd_4 | qtd_5 | faixa_desempenho |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| CGOF | 52 | 4.12 | 2 | 5 | 0 | 3 | 8 | 28 | 13 | Alto desempenho |
| COGEO | 38 | 3.84 | 1 | 5 | 1 | 2 | 12 | 18 | 5 | Alto desempenho |
| CGOV | 45 | 3.51 | 2 | 5 | 0 | 4 | 15 | 21 | 5 | Alto desempenho |
| AUDIT | 18 | 3.00 | 3 | 3 | 0 | 0 | 18 | 0 | 0 | Adequado |
| DIREC | 12 | 2.42 | 1 | 4 | 2 | 4 | 4 | 2 | 0 | Inadequado |

**Leitura caso a caso:**

**CGOF — Alta desempenho (4.12):** média robusta com base em 52 avaliações. A distribuição mostra que a maior parte das avaliações se concentra em notas 4 e 5, com apenas 3 avaliações na nota 2. Unidade com padrão de avaliação diferenciado e uso ativo da escala (notas variam de 2 a 5).

**AUDIT — Adequado (3.00), mas com alerta:** a média 3.00 parece razoável, mas `nota_minima = nota_maxima = 3` indica que **todas as 18 avaliações receberam a mesma nota**. Isso é um sinal de uso pro forma do instrumento — a avaliação está sendo preenchida, mas sem diferenciação de desempenho. Não indica necessariamente problema de resultado; indica problema de processo avaliativo.

**DIREC — Inadequado (2.42):** média abaixo de 2.5 com avaliações nas faixas inferiores da escala (2 notas "Não executado" + 4 notas "Inadequado" de um total de 12). Esta unidade merece atenção prioritária: os próprios atores do sistema reconheceram desempenho insuficiente nos Planos de Trabalho.

### Combinação com outros indicadores

O I09 não deve ser lido isoladamente. Recomenda-se sempre cruzar com:

- **I02 (taxa de cumprimento):** uma unidade com I09 baixo e I02 alto — ou seja, entregas formalmente concluídas mas avaliações baixas — indica que houve cumprimento de metas sem percepção de qualidade. Investigar se os critérios de conclusão estão adequados.
- **I12 (coerência PT vs PE):** uma unidade com I09 alto (PT bem avaliados) e I12 divergente (PE mal avaliado) pode indicar avaliação por cordialidade nos PTs — os gestores avaliam bem individualmente mas reconhecem, no nível coletivo, que os resultados ficaram aquém.
- **`total_avaliacoes_pt`:** unidades com poucas avaliações (abaixo de 5) têm resultados estatisticamente frágeis. Uma média de 5.0 com 2 avaliações não é comparável a uma média de 4.5 com 60 avaliações.

### O que fazer quando o resultado estiver vazio

Se a consulta retornar zero linhas, as causas mais prováveis são:

1. **O ciclo de avaliação não foi concluído no período.** Verifique na Consulta 3 do mapeamento do Eixo 4 se há registros em `avaliacoes` com `plano_trabalho_consolidacao_id` preenchido.
2. **O campo `plano_trabalho_consolidacao_id` não é o identificador correto** nesta versão do PETRVS. Execute a Consulta 4 do mapeamento para verificar.
3. **O período de análise está fora da faixa dos dados.** Verifique as datas dos planos de trabalho no banco antes de ajustar `data_inicio` e `data_fim`.
