# I03 — Taxa de Cumprimento de Metas por Entrega

## 1. Finalidade do indicador

O I03 detalha o que o I02 resume: em vez de uma taxa única por unidade, ele mostra **quanto cada entrega individualmente atingiu em relação à sua própria meta**.

A pergunta central que o I03 responde é: **"Para cada entrega planejada no período, qual foi o percentual da meta efetivamente executado?"**

Se o I02 diz "a CGOF cumpriu 94% das suas entregas", o I03 diz "a entrega X da CGOF atingiu 120% da meta; a entrega Y atingiu 85%; a entrega Z atingiu 0%". É o zoom analítico que permite identificar quais entregas específicas explicam o desempenho geral de uma unidade.

O resultado é uma linha por entrega com a meta planejada, a meta executada, a taxa de atingimento calculada e um status de execução em cinco categorias:

| Status | Condição | Interpretação |
| --- | --- | --- |
| Superexecutada | `taxa_atingimento_perc > 100` | Meta superada — pode indicar excelência ou meta subestimada |
| Concluída | `taxa_atingimento_perc = 100` | Meta atingida exatamente |
| Parcialmente cumprida | `taxa_atingimento_perc >= 70` | Entre 70% e 99% da meta executados |
| Em andamento | `taxa_atingimento_perc > 0` | Execução iniciada, mas abaixo de 70% |
| Não executada | `taxa_atingimento_perc = 0` | Nenhum progresso registrado |

> **Nota sobre reutilização de código:** o bloco `parametros` e o padrão de join desta consulta seguem o mesmo modelo do I02 (`FROM planos_entregas pe JOIN planos_entregas_entregas pee ... JOIN unidades u ON u.id = pe.unidade_id`). A diferença está no escopo de datas: o I02 filtra por sobreposição do ciclo PE (`pe.data_inicio/pe.data_fim`), enquanto o I03 filtra por prazo individual da entrega (`pee.data_fim`). Isso é intencional — o I03 responde "como foi cada entrega que venceu neste período?", não "quais entregas pertencem a um PE que se sobrepõe a este período?". O `entregas_base` aqui tem ainda o campo `nome_entrega`, necessário para identificar cada entrega no resultado.

## 2. Consulta completa (MySQL)

```sql
with parametros as (
    select
        date('2025-01-01') as data_inicio,
        date('2025-12-31') as data_fim,
        0                  as incluir_excluidos
),
entregas_base as (
    select
        u.sigla                                  as unidade_sigla,
        pee.id                                   as id_entrega,
        coalesce(
            nullif(trim(pee.descricao), ''),
            nullif(trim(pee.descricao_entrega), ''),
            'N.I.'
        )                                        as nome_entrega,
        pee.progresso_esperado                   as meta_planejada,
        coalesce(pee.progresso_realizado, 0)     as meta_executada
    from planos_entregas pe
    join planos_entregas_entregas pee
        on pee.plano_entrega_id = pe.id
    join unidades u
        on u.id = pe.unidade_id
    cross join parametros p
    where date(pee.data_fim) between p.data_inicio and p.data_fim
      and (p.incluir_excluidos = 1 or pe.deleted_at  is null)
      and (p.incluir_excluidos = 1 or pee.deleted_at is null)
      and pee.progresso_esperado is not null
      and pee.progresso_esperado > 0
),
entregas_com_taxa as (
    select
        unidade_sigla,
        id_entrega,
        nome_entrega,
        meta_planejada,
        meta_executada,
        round(
            meta_executada / nullif(meta_planejada, 0) * 100,
            2
        )                                        as taxa_atingimento_perc
    from entregas_base
)
select
    unidade_sigla,
    id_entrega,
    nome_entrega,
    meta_planejada,
    meta_executada,
    taxa_atingimento_perc,
    case
        when taxa_atingimento_perc >  100 then 'Superexecutada'
        when taxa_atingimento_perc = 100  then 'Concluída'
        when taxa_atingimento_perc >= 70  then 'Parcialmente cumprida'
        when taxa_atingimento_perc >  0   then 'Em andamento'
        else                                   'Não executada'
    end as status_entrega
from entregas_com_taxa
order by unidade_sigla, taxa_atingimento_perc desc;
```

## 3. Passos da consulta

A consulta tem três CTEs encadeadas antes do `SELECT` final.

### Passo 1 — Bloco `parametros`: painel de controle

Idêntico ao padrão de todos os indicadores. Ajuste `data_inicio` e `data_fim` para o período desejado:

```sql
parametros as (
    select
        date('2025-01-01') as data_inicio,
        date('2025-12-31') as data_fim,
        0                  as incluir_excluidos
)
```

---

### Passo 2 — Bloco `entregas_base`: preparando os dados das entregas

Segue o mesmo padrão de join do I02 (partindo de `planos_entregas pe`, identificando a unidade por `pe.unidade_id`), com duas diferenças: o campo `nome_entrega` foi adicionado (o I03 opera no nível de entrega individual e precisa nomeá-la), e o escopo de datas usa `pee.data_fim` em vez da sobreposição do ciclo PE — ver nota na Seção 1.

```sql
entregas_base as (
    select
        u.sigla                                  as unidade_sigla,
        pee.id                                   as id_entrega,
        coalesce(
            nullif(trim(pee.descricao), ''),
            nullif(trim(pee.descricao_entrega), ''),
            'N.I.'
        )                                        as nome_entrega,
        pee.progresso_esperado                   as meta_planejada,
        coalesce(pee.progresso_realizado, 0)     as meta_executada
    from planos_entregas pe
    join planos_entregas_entregas pee
        on pee.plano_entrega_id = pe.id
    join unidades u
        on u.id = pe.unidade_id
    cross join parametros p
    where date(pee.data_fim) between p.data_inicio and p.data_fim
      and (p.incluir_excluidos = 1 or pe.deleted_at  is null)
      and (p.incluir_excluidos = 1 or pee.deleted_at is null)
      and pee.progresso_esperado is not null
      and pee.progresso_esperado > 0
)
```

**Por que a tabela principal é `planos_entregas pe`:** no PETRVS, `pee.unidade_id` identifica a unidade *executora* de cada entrega — que pode ser diferente da unidade *dona* do PE. Partir de `pe` e usar `pe.unidade_id` garante que todas as entregas do plano sejam atribuídas à unidade que o elaborou, não às unidades colaboradoras. O mesmo padrão está documentado em detalhe no Passo 2 do I02.

**O filtro de data do I03:** `date(pee.data_fim) between p.data_inicio and p.data_fim` seleciona apenas as entregas cujo *prazo individual* vence no período configurado. Isso faz o I03 responder "como foi cada entrega que devia estar concluída neste período?" — um subconjunto do total do ciclo. Para ver todas as entregas de um PE, expanda o período para cobrir o ciclo completo.

**O padrão de nome da entrega:**

```sql
coalesce(
    nullif(trim(pee.descricao), ''),
    nullif(trim(pee.descricao_entrega), ''),
    'N.I.'
) as nome_entrega
```

No banco PETRVS, o nome de uma entrega pode estar em dois campos distintos conforme a versão do sistema. A expressão tenta primeiro `descricao`; se estiver vazio ou nulo, tenta `descricao_entrega`; se ambos estiverem vazios, usa `'N.I.'`. O `trim` remove espaços em branco que fariam um campo parecer preenchido quando na prática está vazio. Esse padrão está documentado em detalhe no arquivo [06.2-eixo2.md](06.2-eixo2.md).

---

### Passo 3 — Bloco `entregas_com_taxa`: calculando a taxa de atingimento

```sql
entregas_com_taxa as (
    select
        unidade_sigla,
        id_entrega,
        nome_entrega,
        meta_planejada,
        meta_executada,
        round(
            meta_executada / nullif(meta_planejada, 0) * 100,
            2
        )                                        as taxa_atingimento_perc
    from entregas_base
)
```

Este bloco calcula a taxa de atingimento de cada entrega individualmente — sem agregação, sem agrupamento.

**A fórmula:**

```sql
meta_executada / nullif(meta_planejada, 0) * 100
```

Divide o que foi realizado pelo que foi planejado e multiplica por 100. O `nullif(meta_planejada, 0)` protege contra divisão por zero — embora o filtro `progresso_esperado > 0` no bloco anterior já exclua essas entregas, o `nullif` é uma proteção adicional caso o filtro seja removido acidentalmente.

**Por que um CTE separado para o cálculo:** separar o cálculo da taxa em um bloco próprio permite que o `SELECT` final use `taxa_atingimento_perc` como alias já calculado dentro do `case when` — sem precisar repetir a fórmula completa dentro da classificação de status.

---

### Passo 4 — SELECT final: classificando o status de cada entrega

```sql
select
    unidade_sigla,
    id_entrega,
    nome_entrega,
    meta_planejada,
    meta_executada,
    taxa_atingimento_perc,
    case
        when taxa_atingimento_perc >  100 then 'Superexecutada'
        when taxa_atingimento_perc = 100  then 'Concluída'
        when taxa_atingimento_perc >= 70  then 'Parcialmente cumprida'
        when taxa_atingimento_perc >  0   then 'Em andamento'
        else                                   'Não executada'
    end as status_entrega
from entregas_com_taxa
order by unidade_sigla, taxa_atingimento_perc desc;
```

**A lógica do `case when`:**

O MySQL avalia as condições de cima para baixo e retorna ao primeiro `then` verdadeiro. A ordem importa:

1. `> 100` é verificado antes de `= 100` para capturar a superexecução antes da conclusão exata.
2. `= 100` captura exatamente 100.00 (após o `ROUND` de 2 casas decimais).
3. `>= 70` captura o intervalo 70–99.99 — entregas que avançaram substancialmente mas não concluíram.
4. `> 0` captura qualquer execução abaixo de 70% — entregas iniciadas mas longe da meta.
5. `else` captura `taxa_atingimento_perc = 0` e também o caso improvável de `NULL`.

**O limiar de 70% para "Parcialmente cumprida":** esse valor segue a mesma lógica dos grupos B/C do I02 (grupos de 70% a 89% e 50% a 69%). Uma entrega que atingiu 70% ou mais completou a maior parte do que foi planejado e está claramente em trajetória de conclusão — o risco gerencial é diferente de uma entrega que está em 20%.

**A ordenação:** dentro de cada unidade, as entregas são apresentadas da maior para a menor taxa, facilitando a identificação rápida das entregas melhor e pior posicionadas.

## 4. Como interpretar o resultado

### Estrutura do resultado

| Coluna | O que informa |
| --- | --- |
| `unidade_sigla` | Sigla da unidade responsável pela entrega |
| `id_entrega` | Identificador único da entrega no banco — útil para rastreabilidade |
| `nome_entrega` | Descrição da entrega conforme registrado no PETRVS |
| `meta_planejada` | Valor de `progresso_esperado` — o que deveria ser atingido |
| `meta_executada` | Valor de `progresso_realizado` — o que foi declarado como executado |
| `taxa_atingimento_perc` | `meta_executada / meta_planejada × 100` — o indicador central desta consulta |
| `status_entrega` | Classificação qualitativa em 5 categorias |

### Exemplo com dados do ICMBio

Resultado hipotético das entregas de Q1/2025 para três unidades, mostrando a distribuição de status:

| unidade_sigla | nome_entrega | meta_planejada | meta_executada | taxa_atingimento_perc | status_entrega |
| --- | --- | --- | --- | --- | --- |
| CGOF | Relatório de gestão Q1 | 1 | 1 | 100.00 | Concluída |
| CGOF | Revisão de normativos | 3 | 4 | 133.33 | Superexecutada |
| CGOF | Capacitações realizadas | 10 | 7 | 70.00 | Parcialmente cumprida |
| CGOV | Mapeamento de riscos | 1 | 0 | 0.00 | Não executada |
| CGOV | Indicadores atualizados | 12 | 9 | 75.00 | Parcialmente cumprida |
| DIREC | Reuniões de alinhamento | 4 | 1 | 25.00 | Em andamento |
| DIREC | Plano de ação enviado | 1 | 0 | 0.00 | Não executada |

**Leitura caso a caso:**

**CGOF — Revisão de normativos (133.33%, Superexecutada):** a unidade planejou revisar 3 normativos e revisou 4. Isso é positivo na execução, mas sinaliza para o gestor uma questão de planejamento: a meta foi subestimada. No I04, essa entrega contribuirá com um score de 133% — elevando a média da unidade acima de 100%. Se várias entregas da CGOF seguirem esse padrão, o I04 revelará uma tendência sistemática de subestimação de metas.

**CGOV — Mapeamento de riscos (0%, Não executada):** nenhum progresso foi registrado. Isso pode significar que o trabalho realmente não foi iniciado, ou que o registro de progresso não foi atualizado no sistema. Antes de concluir sobre falha de execução, o gestor deve verificar se o campo `progresso_realizado` foi preenchido — ausência de registro e ausência de execução são situações distintas, mas o I03 não pode distinguir entre elas.

**DIREC — Reuniões de alinhamento (25%, Em andamento):** a unidade realizou 1 das 4 reuniões planejadas. Com 25% de atingimento, a entrega está categorizada como "Em andamento" — há execução, mas está muito distante da meta. Combinado com o Plano de ação enviado (0%), o resultado do I03 para a DIREC ilustra o padrão que gerou o grupo D no I02.

### Relação entre o número de linhas do I03 e as colunas do I02

O I03 filtra entregas por `pee.data_fim` — o prazo individual. Por isso, para a mesma unidade e o mesmo período, o número de linhas retornado pelo I03 corresponde à coluna `total_vence_no_periodo` do I02, e não a `total_no_ciclo`.

Exemplo com APA-BALEFRA (período 2025-01-01 a 2025-12-31): o I02 retorna `total_no_ciclo = 65` e `total_vence_no_periodo = 36`. Rodando o I03 para o mesmo período, aparecerão **36 linhas** para a APA-BALEFRA — as 29 entregas do PE 884 com prazo em 2026 não aparecem porque o `pee.data_fim` delas está fora do período. Para vê-las no I03, ajuste `data_fim` para `2026-03-31` (cobrindo o ciclo completo desse PE).

### Nuances importantes

**O I03 não valida o progresso — apenas o mede:** os valores de `meta_planejada` e `meta_executada` são inseridos pelos próprios usuários no PETRVS. Uma entrega com taxa de 150% pode indicar trabalho excepcional ou uma meta que foi definida com folga excessiva. O I03 reporta o que foi *declarado*, não o que foi *verificado externamente*.

**Unidades de medida diferentes dentro do resultado:** entregas são medidas em unidades heterogêneas — uma entrega pode ter meta "1 relatório", outra "20 vistorias", outra "4 capacitações". A `taxa_atingimento_perc` padroniza essa comparação em percentual, mas não faz sentido comparar diretamente `meta_planejada = 1` com `meta_planejada = 200` pelo valor absoluto.

**O filtro de 70% para "Parcialmente cumprida" pode ser ajustado:** dependendo do contexto da unidade e da maturidade do planejamento, o gestor pode querer usar 80% como limiar para "Parcialmente cumprida" e reservar "Em andamento" para execuções abaixo disso. Isso é uma decisão analítica — ajuste o valor no `case when` conforme necessário.

**Superexecutadas merecem atenção especial na CGOV:** uma entrega com taxa muito acima de 100% (ex.: 300%, 500%) é sinal de meta inadequada. Se identificadas em quantidade, sugerem que o processo de planejamento da unidade precisa de revisão — não que a execução foi excepcional.

### Leitura integrada com I02 e I04

O I03 é o nível de detalhe natural após o I02. Os usos mais comuns são:

| Ponto de partida no I02 | Uso do I03 |
| --- | --- |
| Unidade no grupo D (taxa < 50%) | Identificar quais entregas específicas não foram concluídas — foco da conversa gerencial |
| Unidade no grupo A com I04 muito acima de 100% | Listar as entregas "Superexecutadas" — candidatas a revisão de meta no próximo ciclo |
| Qualquer unidade com base pequena (< 10 entregas) | Verificar se o grupo D ou A não é artefato de uma ou duas entregas atípicas |

O I04 complementa o I03 ao agregar todas as taxas individuais em uma única média ponderada por unidade — substituindo a visão entrega-a-entrega por uma nota sintética de desempenho.
